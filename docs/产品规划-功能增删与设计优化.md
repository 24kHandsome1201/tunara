# Conduit 产品规划：功能增删与设计优化

> 基于 M0-M3 完成后的全量代码审读，识别当前产品的焦点问题、应砍/应优化/应新增的功能，以及设计品味层面的方向判断。

## 一、现状快照

Conduit 是一个 AI 原生终端（Tauri v2 + React 19 + Zustand + xterm.js），当前产品边界是“带智能侧栏的真实终端”：

```
┌─────────────┬───────────────────────┬──────────────┐
│  Sidebar    │     MainArea          │  DiffPanel   │
│  会话列表    │     xterm 终端         │  Git 改动     │
│  按目录分组   │     + 底部状态栏       │  只读 Review   │
└─────────────┴───────────────────────┴──────────────┘
```

**已完成能力：**
- 多 PTY 会话管理（tab + sidebar 双重导航）
- 11 种 AI agent CLI 自动检测与品牌标识
- Git 读操作（git2）；commit/push/stage 不作为主 invoke 能力暴露
- DiffPanel 已收口为只读 review 面板，不暴露 stage/commit/push GUI
- 设置面板（主题/强调色/光标/字号/终端 CLI 路径检测）
- macOS 毛玻璃、深色模式、窗口状态持久化
- OSC 7/133 shell integration（cwd 追踪 + agent 退出检测）

**代码规模：** ~2500 行前端 TypeScript + ~2000 行 Rust 后端。

## 二、应该砍掉的

### 2.1 DiffPanel 的 commit/push 操作 UI（已砍）

**旧问题：** DiffPanel 曾计划包含 textarea（commit message）、提交按钮、提交并推送按钮、全选/部分 stage 复选框，约 80 行代码 + 大量状态变量（`commitMsg`、`busy`、`feedback`、`staged`）。

**当前状态：** 主 UI 已改为只读 review 面板。主 invoke 面也不再暴露 `gitCommit` / `gitPush`；用户需要写操作时应回到真实终端里的 git CLI，Conduit 只负责展示 review 上下文。

**为什么砍：**

1. **与产品定位冲突。** Conduit 的核心叙事是"看 agent 在做什么、review agent 改了什么"。用户在这个终端里跑 Claude Code / Codex，agent 自己会 commit。手动写 commit message 是一个完全不同的使用场景——这是在重造半个 Git GUI。
2. **交互质量不可能做好。** 一个 300px 宽的面板里塞 textarea + 两个按钮 + staged 计数 + remote 状态，信息密度过高。如果要做到 VS Code Source Control 的水平，需要的 UI 面积和交互复杂度远超这个侧栏能承载的。
3. **状态管理负担。** `staged` Set、`busy`、`feedback`、`commitMsg` 四个 state 加 `handleCommit` async 函数，让一个本应只读的面板变成了最复杂的组件。

**砍成什么：** DiffPanel 变为**只读 Review 面板**（详见 §3.2）。

**涉及文件：**
- `src/ui/DiffPanel.tsx`：删除 textarea、commit/push 按钮、staged 相关逻辑
- `src/modules/git/git-bridge.ts`：只保留 `gitStatus` / `gitDiff` / `gitAheadBehind` 这类读接口
- `src-tauri/src/modules/git/commit.rs`：仅保留显式 pathspec 的单元测试夹具，避免历史写路径回流到运行时

### 2.2 NotifCenter 空模块

**现状：** `src/ui/NotifCenter.tsx` 只有一行 `export {}`。

**操作：** 直接删除文件。如果未来需要通知机制，从零开始设计比保留空壳更干净。

## 三、应该优化的（现有功能做得更好）

### 3.1 侧栏标题推导机制

**现状问题：**

`TerminalView.tsx` 通过 xterm `onData` 回调逐字符拼接 `inputBuffer`，遇到回车时提交到 `onCommandDetected`，经 `cleanTerminalText` 清理 ESC 序列后作为 `session.lastCommand` 显示在侧栏。

这个方案的脆弱点：
- **Multi-byte 输入**（中文 IME、粘贴）会导致 buffer 错位
- **Readline 编辑**（Ctrl+A 跳行首、Alt+B 跳词）产生的 CSI 序列被 `cleanTerminalText` 清理后可能截断命令
- **历史命令导航**（上下箭头）会触发多次伪命令
- 整个 keystroke parsing 逻辑约 30 行，是 TerminalView 中最容易出 bug 的部分

**优化方向：依赖 Shell Integration 而非 Keystroke Parsing**

现代 shell 的 OSC 133 protocol 已经包含了命令内容：

```
OSC 133;A ST  → prompt 开始
OSC 133;B ST  → prompt 结束（用户输入开始）
OSC 133;C ST  → 命令执行开始（此时 shell 已解析完整命令行）
OSC 133;D;N ST → 命令结束（N = exit code）
```

当前代码已经注册了 OSC 133;A handler（用于 agent 退出检测）。扩展它来捕获命令内容：

1. 监听 OSC 133;C，从 B→C 之间的 xterm buffer 行提取完整命令（shell 已经处理好所有编辑和展开）
2. 降级方案：对不支持 OSC 133 的 shell，保留当前 keystroke 逻辑作 fallback

**好处：**
- 命令内容来自 shell 而非 UI 层拼接，准确度从 ~85% 提升到 ~99%
- 减少 TerminalView 中的 keystroke parsing 代码
- 副产品：能拿到命令的 exit code，用于侧栏显示成功/失败状态

**涉及文件：**
- `src/ui/TerminalView.tsx`：扩展 OSC 133 handler，简化 onData 中的 inputBuffer 逻辑
- `src/ui/types.ts`：Session 可增加 `lastExitCode?: number`

### 3.2 DiffPanel → Review 面板

**将 DiffPanel 从 git commit 工具变成 agent 工作的审查面板。**

**新设计：**

```
┌─────────────────────────┐
│ 改动  ⎇ main  3 文件     │  ← 标题栏保留
├─────────────────────────┤
│ □ M  src/app/App.tsx     │  ← 文件列表（只读，无 checkbox）
│   +12 −3               │
│ ┌─────────────────────┐ │  ← 展开 diff（保留）
│ │ - old line          │ │
│ │ + new line          │ │
│ └─────────────────────┘ │
│ A  src/ui/NewFile.tsx    │
│   +45 −0               │
├─────────────────────────┤
│  origin/main · ↑2 ↓0   │  ← 底部：只保留 remote 状态信息
└─────────────────────────┘
```

**删除的元素：**
- Stage/unstage 复选框
- Commit message textarea
- 提交 / 提交并推送 按钮
- `staged` state、`commitMsg` state、`busy` state、`feedback` state
- `handleCommit` 函数

**保留的元素：**
- 文件列表 + 状态 badge (M/A/D/R)
- 点击展开 inline diff
- 分支名 + ahead/behind 信息
- Empty 状态（工作区干净 / 非 Git 仓库 / 加载中）

**新增的元素：**
- agent 运行结束后自动刷新 diff（当前需要 `gitNonce` 手动触发，应该在 OSC 133;D 时自动 bump）
- 文件改动的 summary 更突出（当前 summary 在标题栏右侧太小）

**涉及文件：**
- `src/ui/DiffPanel.tsx`：重构，删除写操作 UI
- `src/state/sessions.ts`：`refreshGit` 可以在 agent 退出事件中自动调用

### 3.3 Session 模型补全生命周期

**现状问题：**

`Session` interface 缺少运行状态。一个 shell session 不知道自己是 idle 还是有进程在跑：

```typescript
// 当前 Session — 没有 runState
export interface Session {
  id: string;
  agent?: AgentCode;
  title: string;
  dir: string;
  branch: string;
  lastCommand?: string;
  shellTitle?: string;
  changes?: { ... };
  updatedAt: number;
}
```

**缺失导致的问题：**
1. 侧栏无法显示 running indicator（哪个 session 正在跑 agent）
2. 关闭 session 时无法警告"进程还在运行"
3. 无法做"agent 跑完自动打开 diff"的联动
4. StatusBar 的 agent 名称是 static 的，不知道 agent 是在跑还是已退出

**优化方案：**

```typescript
export interface Session {
  id: string;
  agent?: AgentCode;
  title: string;
  dir: string;
  branch: string;

  // 生命周期
  runState: "idle" | "running" | "done" | "failed";
  startedAt?: number;
  completedAt?: number;

  // 动态标题源
  lastCommand?: string;
  lastExitCode?: number;
  shellTitle?: string;

  // git 改动
  changes?: { ... };

  updatedAt: number;
}
```

**状态转移：**

```
新建 session → idle
输入命令 → running（onCommandDetected / onAgentDetected）
命令/agent 结束 → done（OSC 133;D 或 onAgentExited）
PTY 退出 → idle（onExit callback）
```

**UI 影响：**
- SessionCard：running 状态显示一个微弱的脉动动画（当前已有 `pulseDot` keyframe）
- SessionCard：done + 有 changes 时，subtitle 里显示改动 summary
- StatusBar：agent running 时名称旁加一个 `·` 脉动点

**涉及文件：**
- `src/ui/types.ts`：Session interface 扩展
- `src/state/sessions.ts`：createSession 默认 `runState: "idle"`
- `src/ui/TerminalView.tsx`：在 OSC 133;C/D handler 中触发状态变更
- `src/ui/SessionCard.tsx`：根据 runState 调整视觉
- `src/app/App.tsx`：onAgentDetected / onAgentExited 中更新 runState

### 3.4 样式系统：高频 inline style 提取为 CSS 类

**现状问题：**

组件中 100% 使用 inline style，导致：
- 大量重复模式（`fontSize: "var(--fs-body)"`, `fontFamily: "var(--font-mono)"` 在全项目出现 30+ 次）
- 无法使用 CSS 伪类（`:focus-visible`、`:first-child`、`::placeholder`）
- 无法使用 CSS 过渡和动画（当前的 `transition` 写在 inline style 里，和 hover 效果断裂）
- 代码密度高，一个按钮的 style 经常超过 5 行

**优化方案：不改架构，只提取高频模式**

在 `tokens.css` 底部追加通用工具类：

```css
/* 文本 */
.text-mono    { font-family: var(--font-mono); }
.text-body    { font-size: var(--fs-body); color: var(--c-text-primary); }
.text-meta    { font-size: var(--fs-meta); color: var(--c-text-5); font-family: var(--font-mono); }
.text-badge   { font-size: var(--fs-badge); font-weight: 700; font-family: var(--font-mono); }
.text-ellipsis { overflow: hidden; text-overflow: ellipsis; white-space: nowrap; }

/* 布局 */
.flex-center  { display: flex; align-items: center; justify-content: center; }
.flex-col     { display: flex; flex-direction: column; }

/* 按钮 */
.btn-icon     { width: 24px; height: 24px; border-radius: var(--r-btn); border: none;
                background: transparent; cursor: pointer; display: flex;
                align-items: center; justify-content: center; }

/* 面板 */
.panel-header { height: var(--h-titlebar); border-bottom: 1px solid var(--c-border-1);
                display: flex; align-items: center; padding: 0 14px; gap: 8px; flex-shrink: 0; }
```

**不做的事：**
- 不引入 CSS-in-JS 库（Tailwind / styled-components）
- 不改现有 inline style 的架构——只在新增/修改组件时逐步迁移
- 不强制所有组件一次性改完

**涉及文件：**
- `src/styles/tokens.css`：追加工具类
- 各组件文件：在修改时逐步替换重复 inline style

## 四、应该新增的功能

### 4.1 P0：Session 分栏（Split Pane）

**为什么是 P0：**

AI 终端最核心的使用场景是：一边看 agent 跑，一边在另一个 pane 里手动操作（查资料、手动测试、看日志）。当前只有 tab 切换，用户必须在 agent 输出和自己的操作之间反复切换——这直接伤害了"同时看到 agent 在做什么"的核心价值。

**设计方案：**

```
┌──────────────────────────────────────┐
│  Tab1  │ Tab2  │ +                   │  ← Titlebar
├────────┴───────┴─────────────────────┤
│                 │                    │
│  Pane A         │  Pane B            │
│  (agent 在跑)   │  (我在操作)         │
│                 │                    │
│                 │                    │
├─────────────────┴────────────────────┤
│  ~/code/project · ⎇ main · CC       │  ← StatusBar
└──────────────────────────────────────┘
```

**交互设计：**

- `⌘D` 水平分栏（左右）：把当前终端区域一分为二，右侧新建一个 session
- `⌘⇧D` 垂直分栏（上下）：同理
- 只支持一次 split（最多 2 个 pane）。不做递归 tile layout——复杂度爆炸，且实际需求极少
- 分栏线可拖拽调整比例
- `⌘]` / `⌘[` 在 pane 间切换焦点
- 关闭一个 pane 后自动回到单 pane

**状态模型：**

```typescript
interface SplitState {
  mode: "single" | "horizontal" | "vertical";
  paneA: string;  // sessionId
  paneB: string | null;  // sessionId
  ratio: number;  // 0.3 ~ 0.7, 默认 0.5
}
```

Split 状态属于 UI store（`ui.ts`），不属于 session store。每个 tab 共享同一个 split 布局（切 tab 时 paneA 跟 tab 的 activeSession 走）。

**实现路径：**
1. `MainArea.tsx` 支持渲染两个 TerminalView（当前只渲染一个 active）
2. 加一个 `SplitHandle` 拖拽组件（纯 CSS + mousedown/mousemove）
3. UI store 扩展 split 相关 state 和 action
4. 快捷键注册

**涉及文件：**
- `src/ui/MainArea.tsx`：核心改动，支持双 pane 渲染
- `src/state/ui.ts`：新增 SplitState
- `src/app/App.tsx`：注册 ⌘D / ⌘⇧D / ⌘] / ⌘[ 快捷键
- 新增 `src/ui/SplitHandle.tsx`：拖拽分栏线组件

**工作量估计：** 前端 ~300 行新增/修改。

### 4.2 P1：会话持久化与恢复

**现状问题：**

`persist.ts` 有 `loadSessions` / `saveSessions`，但：
- `saveSessions` 在 `closeSession` 时调用，其他时候不调用
- 关闭 app 后再打开，session 列表和 cwd 全丢
- PTY 进程（和 scrollback）无法跨重启恢复，这是可接受的

**设计方案：**

**持久化什么：**
- Session 列表（id、dir、branch、agent、title、runState）
- 活跃 sessionId
- UI 布局状态（sidebarVisible、panelVisible、splitState）

**不持久化什么：**
- PTY scrollback（太大，且终端内容本身不是关键信息）
- git changes（重启后重新 fetch 即可）
- lastCommand / shellTitle（临时 UI 状态）

**恢复行为：**
1. 启动时读取 persisted sessions
2. 为每个 session 创建新的 PTY，cwd 设为 session.dir
3. 如果 session 有 agent 标记，不自动重启 agent（agent 会话是一次性的）
4. 如果无 persisted sessions，创建默认 `~` 终端（当前行为）

**触发持久化的时机：**
- Session 增删
- cwd 变更
- App 即将关闭（Tauri `close-requested` 事件）
- 定时（每 30s）作为兜底

**涉及文件：**
- `src/state/persist.ts`：完善 save/load 逻辑
- `src/app/App.tsx`：在关键时机触发 save；启动恢复流程改进
- `src/state/ui.ts`：UI 布局状态也持久化（可以复用 localStorage 现有机制）

**工作量估计：** ~150 行修改。

### 4.3 P2：Command Palette

**为什么加：**

当前快捷键只有 4 个（⌘T/⌘W/⌘,/⌘\），硬编码在 App.tsx。随着功能增长，需要一个可发现的入口。Command Palette 是有 taste 的终端 app 的标配（VS Code ⌘⇧P、Warp ⌘K、Raycast ⌘Space）。

**设计方案：**

```
┌──────────────────────────────────┐
│ 🔍 输入命令或搜索...              │
├──────────────────────────────────┤
│ 最近                             │
│   切换到: Claude Code · project  │
│   切换到: 终端 · api-server      │
│ 操作                             │
│   新建终端          ⌘T           │
│   新建 Agent 会话   ⌘⇧N          │
│   分栏              ⌘D           │
│   切换侧栏          ⌘\           │
│   切换审查面板       ⌘⇧\          │

```

**触发：** `⌘K`（与 Warp 一致，且不冲突）

**命令列表：**
1. 会话切换（按 fuzzy match 排序）
2. 新建终端 / 新建 Agent 会话（选择 agent 类型后自动启动 CLI）
3. UI 控制（toggle sidebar / panel / split）
4. 设置

**实现路径：**

- 一个 input + 一个 filtered list，<200 行
- 遵循现有 overlay 规范（Escape 关闭、backdrop blur、sheetIn 动画）

**涉及文件：**
- 新增 `src/ui/overlays/CommandPalette.tsx`
- `src/ui/types.ts`：OverlayType 扩展为 `null | "settings" | "command-palette"`
- `src/app/App.tsx`：注册 ⌘K 快捷键

**工作量估计：** ~200 行新增。

### 4.4 P3：Agent 状态条（Thin Overlay）

**现状问题：**

Agent 的输出和普通 shell 共用同一个 xterm。Claude Code 的 markdown 输出、tool 调用指示、思考过程全是 raw terminal text。用户必须盯着滚动的终端才知道 agent 在干什么——而 agent 的输出速度远快于人类阅读速度。

**设计方案：**

在 agent 运行时，终端顶部浮一个 32px 的状态条：

```
┌──────────────────────────────────────┐
│ 🟡 Claude Code · 运行中 · 已编辑 3 文件 │  ← Agent 状态条
├──────────────────────────────────────┤
│                                      │
│  (xterm 正常渲染 agent 输出)           │
│                                      │
└──────────────────────────────────────┘
```

**信息来源：**
- Agent 名称：`session.agent`
- 运行状态：`session.runState`
- 文件改动计数：`session.changes?.files.length`（通过 gitNonce 定时刷新）

**不做的事：**
- 不解析 agent 的 stdout 来提取结构化事件（这需要为每个 agent 写 parser，维护成本太高）
- 不做完整的 transcript view（这是另一个产品级别的功能）
- 只在有 agent 运行时显示，agent 退出后 3 秒淡出

**实现路径：**
- 在 `TerminalView.tsx` 外层包一个 overlay div
- 信息来自 session store，不需要新的数据源
- 约 60 行新增

**涉及文件：**
- `src/ui/TerminalView.tsx` 或新增 `src/ui/AgentStatusBar.tsx`
- `src/ui/MainArea.tsx`：在 TerminalView 上层渲染 AgentStatusBar

**工作量估计：** ~80 行新增。

## 五、设计品味：核心叙事

Conduit 当前的问题不是做得太少，而是**焦点分散**。它同时在做终端、Git GUI、Agent 检测、设置面板——每一个都做了 70 分，但没有一个做到 90 分。

**有 taste 的做法是砍到只剩核心叙事：**

> "我能同时看到 agent 在做什么，和它改了什么。"

具体对应：

```
左 pane：agent 在跑          右 pane：我在操作
              ↓
      右栏：只读 Review 面板（文件改动 + diff）
```

**优先级排序的逻辑：**

| 优先级 | 功能 | 解决的问题 |
|--------|------|-----------|
| P0 | Split Pane | "我不能同时看 agent 和做自己的事" |
| P0 | DiffPanel → Review（砍 commit UI） | "这个面板又能看又能操作，什么都做不好" |
| P0 | Session 生命周期 | "我不知道 agent 还在不在跑" |
| P1 | 会话持久化 | "关掉 app 重开一切都没了" |
| P1 | 侧栏标题推导优化 | "侧栏标题经常显示错误命令" |
| P2 | Command Palette | "功能越来越多，我找不到入口" |
| P2 | CSS 类提取 | 可维护性，无直接用户感知 |
| P3 | Agent 状态条 | "agent 跑太快我看不过来" |

**每个阶段完成后的交付物：**

- **Phase 1（P0）：** Split Pane + Review 面板 + Session 生命周期 → Conduit 成为“带智能侧栏的真实终端”，而不是 agent 工作台
- **Phase 2（P1）：** 持久化 + 标题优化 → 日常使用不再有割裂感
- **Phase 3（P2-P3）：** Command Palette + Agent 状态条 + CSS 清理 → 打磨细节

---

## 六、不做清单

以下功能明确不在规划范围内，记录理由避免后续重新讨论：

| 不做的事 | 理由 |
|----------|------|
| Agent stdout 结构化解析 | 需要为每个 agent 写 parser（至少 11 种），维护成本远超收益 |
| 内置 Agent SDK 调用 | 决策 D3 已确定走 CLI 而非 SDK，不变 |
| 完整的 terminal multiplexer（tmux 级 tile） | 递归分栏的复杂度不值得，一次 split 覆盖 95% 场景 |
| 内置文件编辑器 | Conduit 不是 IDE，文件编辑交给 agent 或外部编辑器 |
| 远程 SSH 连接 | 目标用户在本机跑 agent，不是运维场景 |
| Tab 拖拽排序 | 当前 tab 数量不大（通常 <8），ROI 不高 |
| 插件系统 | 产品太早期，抽象层会变成负担 |
