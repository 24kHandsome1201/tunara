# Tunara 产品规划：功能增删与设计优化

> 基于 M0-M3 完成后的全量代码审读，识别当前产品的焦点问题、应砍/应优化/应新增的功能，以及设计品味层面的方向判断。

## 一、现状快照

Tunara 是一个 AI 原生终端（Tauri v2 + React 19 + Zustand + xterm.js），当前产品边界是“带智能侧栏的真实终端”：

```
┌─────────────┬───────────────────────┬──────────────┐
│  Sidebar    │     MainArea          │  DiffPanel   │
│  会话列表    │     xterm 终端         │  Git 改动     │
│  按目录分组   │     + 底部状态栏       │  只读 Review   │
└─────────────┴───────────────────────┴──────────────┘
```

**已完成能力：**
- 多 PTY 会话管理（tab + sidebar 双重导航）
- 11 种 AI agent CLI 自动检测与品牌标识
- Git 读操作（git2）；commit/push/stage 不作为主 invoke 能力暴露
- DiffPanel 已收口为只读 review 面板，不暴露 stage/commit/push GUI
- 设置面板（主题/强调色/光标/字号/终端 CLI 路径检测）
- macOS 毛玻璃、深色模式、窗口状态持久化
- OSC 7/133 shell integration（cwd 追踪 + agent 退出检测）

**代码规模：** ~2500 行前端 TypeScript + ~2000 行 Rust 后端。

## 二、应该砍掉的

### 2.1 DiffPanel 的 commit/push 操作 UI（已砍）

**旧问题：** DiffPanel 曾计划包含 textarea（commit message）、提交按钮、提交并推送按钮、全选/部分 stage 复选框，约 80 行代码 + 大量状态变量（`commitMsg`、`busy`、`feedback`、`staged`）。

**当前状态：** 主 UI 已改为只读 review 面板。主 invoke 面也不再暴露 `gitCommit` / `gitPush`；用户需要写操作时应回到真实终端里的 git CLI，Tunara 只负责展示 review 上下文。

**为什么砍：**

1. **与产品定位冲突。** Tunara 的核心叙事是"看 agent 在做什么、review agent 改了什么"。用户在这个终端里跑 Claude Code / Codex，agent 自己会 commit。手动写 commit message 是一个完全不同的使用场景——这是在重造半个 Git GUI。
2. **交互质量不可能做好。** 一个 300px 宽的面板里塞 textarea + 两个按钮 + staged 计数 + remote 状态，信息密度过高。如果要做到 VS Code Source Control 的水平，需要的 UI 面积和交互复杂度远超这个侧栏能承载的。
3. **状态管理负担。** `staged` Set、`busy`、`feedback`、`commitMsg` 四个 state 加 `handleCommit` async 函数，让一个本应只读的面板变成了最复杂的组件。

**砍成什么：** DiffPanel 变为**只读 Review 面板**（详见 §3.2）。

**涉及文件：**
- `src/ui/DiffPanel.tsx`：删除 textarea、commit/push 按钮、staged 相关逻辑
- `src/modules/git/git-bridge.ts`：只保留 `gitStatus` / `gitDiff` / `gitAheadBehind` 这类读接口
- `src-tauri/src/modules/git/commit.rs`：仅保留显式 pathspec 的单元测试夹具，避免历史写路径回流到运行时

### 2.2 NotifCenter 空模块

**现状：** `src/ui/NotifCenter.tsx` 只有一行 `export {}`。

**操作：** 直接删除文件。如果未来需要通知机制，从零开始设计比保留空壳更干净。

## 三、应该优化的（现有功能做得更好）

### 3.1 侧栏标题推导机制

**现状问题：**

`TerminalView.tsx` 通过 xterm `onData` 回调逐字符拼接 `inputBuffer`，遇到回车时提交到 `onCommandDetected`，经 `cleanTerminalText` 清理 ESC 序列后作为 `session.lastCommand` 显示在侧栏。

这个方案的脆弱点：
- **Multi-byte 输入**（中文 IME、粘贴）会导致 buffer 错位
- **Readline 编辑**（Ctrl+A 跳行首、Alt+B 跳词）产生的 CSI 序列被 `cleanTerminalText` 清理后可能截断命令
- **历史命令导航**（上下箭头）会触发多次伪命令
- 整个 keystroke parsing 逻辑约 30 行，是 TerminalView 中最容易出 bug 的部分

**优化方向：依赖 Shell Integration 而非 Keystroke Parsing**

现代 shell 的 OSC 133 protocol 已经包含了命令内容：

```
OSC 133;A ST  → prompt 开始
OSC 133;B ST  → prompt 结束（用户输入开始）
OSC 133;C ST  → 命令执行开始（此时 shell 已解析完整命令行）
OSC 133;D;N ST → 命令结束（N = exit code）
```

当前代码已经注册了 OSC 133;A handler（用于 agent 退出检测）。扩展它来捕获命令内容：

1. 监听 OSC 133;C，从 B→C 之间的 xterm buffer 行提取完整命令（shell 已经处理好所有编辑和展开）
2. 降级方案：对不支持 OSC 133 的 shell，保留当前 keystroke 逻辑作 fallback

**好处：**
- 命令内容来自 shell 而非 UI 层拼接，准确度从 ~85% 提升到 ~99%
- 减少 TerminalView 中的 keystroke parsing 代码
- 副产品：能拿到命令的 exit code，用于侧栏显示成功/失败状态

**涉及文件：**
- `src/ui/TerminalView.tsx`：扩展 OSC 133 handler，简化 onData 中的 inputBuffer 逻辑
- `src/ui/types.ts`：Session 可增加 `lastExitCode?: number`

### 3.2 DiffPanel → Review 面板

**将 DiffPanel 从 git commit 工具变成 agent 工作的审查面板。**

**新设计：**

```
┌─────────────────────────┐
│ 改动  ⎇ main  3 文件     │  ← 标题栏保留
├─────────────────────────┤
│ □ M  src/app/App.tsx     │  ← 文件列表（只读，无 checkbox）
│   +12 −3               │
```

**删除的元素：**

- `staged` state、`commitMsg` state、`busy` state、`feedback` state

**保留的元素：**
- 文件列表 + 状态 badge (M/A/D/R)
- 点击展开 inline diff

**涉及文件：**
- `src/ui/DiffPanel.tsx`：重构，删除写操作 UI
- `src/state/sessions.ts`：`refreshGit` 可以在 agent 退出事件中自动调用

### 3.3 Session 模型补全生命周期

**现状问题：**

`Session` interface 缺少运行状态。一个 shell session 不知道自己是 idle 还是有进程在跑：

```typescript
// 当前 Session — 没有 runState
export interface Session {
  id: string;
  agent?: AgentCode;
  title: string;
  dir: string;
  branch: string;
  lastCommand?: string;
  shellTitle?: string;
  changes?: { ... };
  updatedAt: number;
}
```

**缺失导致的问题：**
1. 侧栏无法显示 running indicator（哪个 session 正在跑 agent）
2. 关闭 session 时无法警告"进程还在运行"
3. 无法做"agent 跑完自动打开 diff"的联动
4. StatusBar 的 agent 名称是 static 的，不知道 agent 是在跑还是已退出

**优化方案：**

```typescript
export interface Session {
  id: string;
  agent?: AgentCode;
  title: string;
  dir: string;
  branch: string;

  // 生命周期
  runState: "idle" | "running" | "done" | "failed";
  startedAt?: number;
  completedAt?: number;

  // 动态标题源
  lastCommand?: string;
  lastExitCode?: number;
  shellTitle?: string;

  // git 改动
  changes?: { ... };

  updatedAt: number;
}
```

**状态转移：**

```
新建 session → idle
输入命令 → running（onCommandDetected / onAgentDetected）
命令/agent 结束 → done（OSC 133;D 或 onAgentExited）
PTY 退出 → idle（onExit callback）
```

**UI 影响：**
- SessionCard：running 状态显示一个微弱的脉动动画（当前已有 `pulseDot` keyframe）
- SessionCard：done + 有 changes 时，subtitle 里显示改动 summary
- StatusBar：agent running 时名称旁加一个 `·` 脉动点

**涉及文件：**
- `src/ui/types.ts`：Session interface 扩展
- `src/state/sessions.ts`：createSession 默认 `runState: "idle"`
- `src/ui/TerminalView.tsx`：在 OSC 133;C/D handler 中触发状态变更
- `src/ui/SessionCard.tsx`：根据 runState 调整视觉
- `src/app/App.tsx`：onAgentDetected / onAgentExited 中更新 runState

### 3.4 样式系统：高频 inline style 提取为 CSS 类

**现状问题：**

组件中 100% 使用 inline style，导致：
- 大量重复模式（`fontSize: "var(--fs-body)"`, `fontFamily: "var(--font-mono)"` 在全项目出现 30+ 次）
- 无法使用 CSS 伪类（`:focus-visible`、`:first-child`、`::placeholder`）
- 无法使用 CSS 过渡和动画（当前的 `transition` 写在 inline style 里，和 hover 效果断裂）
- 代码密度高，一个按钮的 style 经常超过 5 行

**优化方案：不改架构，只提取高频模式**

在 `tokens.css` 底部追加通用工具类：

```css
/* 文本 */
.text-mono    { font-family: var(--font-mono); }
.text-body    { font-size: var(--fs-body); color: var(--c-text-primary); }
.text-meta    { font-size: var(--fs-meta); color: var(--c-text-5); font-family: var(--font-mono); }
.text-badge   { font-size: var(--fs-badge); font-weight: 700; font-family: var(--font-mono); }
.text-ellipsis { overflow: hidden; text-overflow: ellipsis; white-space: nowrap; }

/* 布局 */
.flex-center  { display: flex; align-items: center; justify-content: center; }
.flex-col     { display: flex; flex-direction: column; }

/* 按钮 */
.btn-icon     { width: 24px; height: 24px; border-radius: var(--r-btn); border: none;
                background: transparent; cursor: pointer; display: flex;
                align-items: center; justify-content: center; }

/* 面板 */
.panel-header { height: var(--h-titlebar); border-bottom: 1px solid var(--c-border-1);
                display: flex; align-items: center; padding: 0 14px; gap: 8px; flex-shrink: 0; }
```

**不做的事：**
- 不引入 CSS-in-JS 库（Tailwind / styled-components）
- 不改现有 inline style 的架构——只在新增/修改组件时逐步迁移
- 不强制所有组件一次性改完

**涉及文件：**
- `src/styles/tokens.css`：追加工具类
- 各组件文件：在修改时逐步替换重复 inline style

## 四、应该新增的功能

### 4.1 P0：Session 分栏（Split Pane）

**为什么是 P0：**

AI 终端最核心的使用场景是：一边看 agent 跑，一边在另一个 pane 里手动操作（查资料、手动测试、看日志）。当前只有 tab 切换，用户必须在 agent 输出和自己的操作之间反复切换——这直接伤害了"同时看到 agent 在做什么"的核心价值。

**设计方案：**

```
┌──────────────────────────────────────┐
│  Tab1  │ Tab2  │ +                   │  ← Titlebar
├────────┴───────┴─────────────────────┤
│                 │                    │
│  Pane A         │  Pane B            │
│  (agent 在跑)   │  (我在操作)         │
│                 │                    │
│                 │                    │
├─────────────────┴────────────────────┤
│  ~/code/project · ⎇ main · CC       │  ← StatusBar
└──────────────────────────────────────┘
```

**交互设计：**

- `⌘D` 水平分栏（左右）：把当前终端区域一分为二，右侧新建一个 session
- `⌘⇧D` 垂直分栏（上下）：同理
- 只支持一次 split（最多 2 个 pane）。不做递归 tile layout——复杂度爆炸，且实际需求极少
- 分栏线可拖拽调整比例
- `⌘]` / `⌘[` 在 pane 间切换焦点
- 关闭一个 pane 后自动回到单 pane

**状态模型：**

```typescript
interface SplitState {
  mode: "single" | "horizontal" | "vertical";
  paneA: string;  // sessionId
  paneB: string | null;  // sessionId
  ratio: number;  // 0.3 ~ 0.7, 默认 0.5
}
```

Split 状态属于 UI store（`ui.ts`），不属于 session store。每个 tab 共享同一个 split 布局（切 tab 时 paneA 跟 tab 的 activeSession 走）。

**实现路径：**
1. `MainArea.tsx` 支持渲染两个 TerminalView（当前只渲染一个 active）
2. 加一个 `SplitHandle` 拖拽组件（纯 CSS + mousedown/mousemove）
3. UI store 扩展 split 相关 state 和 action
4. 快捷键注册

**涉及文件：**
- `src/ui/MainArea.tsx`：核心改动，支持双 pane 渲染
- `src/state/ui.ts`：新增 SplitState
- `src/app/App.tsx`：注册 ⌘D / ⌘⇧D / ⌘] / ⌘[ 快捷键
- 新增 `src/ui/SplitHandle.tsx`：拖拽分栏线组件

**工作量估计：** 前端 ~300 行新增/修改。

### 4.2 P1：会话持久化与恢复

**现状问题：**

`persist.ts` 有 `loadSessions` / `saveSessions`，但：
- `saveSessions` 在 `closeSession` 时调用，其他时候不调用
- 关闭 app 后再打开，session 列表和 cwd 全丢
- PTY 进程（和 scrollback）无法跨重启恢复，这是可接受的

**设计方案：**

**持久化什么：**
- Session 列表（id、dir、branch、agent、title、runState）
- 活跃 sessionId
- UI 布局状态（sidebarVisible、panelVisible、splitState）

**不持久化什么：**
- PTY scrollback（太大，且终端内容本身不是关键信息）
- git changes（重启后重新 fetch 即可）
- lastCommand / shellTitle（临时 UI 状态）

**恢复行为：**
1. 启动时读取 persisted sessions
2. 为每个 session 创建新的 PTY，cwd 设为 session.dir
3. 如果 session 有 agent 标记，不自动重启 agent（agent 会话是一次性的）
4. 如果无 persisted sessions，创建默认 `~` 终端（当前行为）

**触发持久化的时机：**
- Session 增删
- cwd 变更
- App 即将关闭（Tauri `close-requested` 事件）
- 定时（每 30s）作为兜底

**涉及文件：**
- `src/state/persist.ts`：完善 save/load 逻辑
- `src/app/App.tsx`：在关键时机触发 save；启动恢复流程改进
- `src/state/ui.ts`：UI 布局状态也持久化（可以复用 localStorage 现有机制）

**工作量估计：** ~150 行修改。

**为什么加：**

当前快捷键只有 4 个（⌘T/⌘W/⌘,/⌘\），硬编码在 App.tsx。随着功能增长，需要一个可发现的入口。Command Palette 是有 taste 的终端 app 的标配（VS Code ⌘⇧P、Warp ⌘K、Raycast ⌘Space）。

**设计方案：**

```
┌──────────────────────────────────┐
│ 🔍 输入命令或搜索...              │
├──────────────────────────────────┤
│ 最近                             │
│   切换到: Claude Code · project  │
│   切换到: 终端 · api-server      │
│ 操作                             │
│   新建终端          ⌘T           │
```

**命令列表：**

**涉及文件：**
- 新增 `src/ui/overlays/CommandPalette.tsx`
- `src/ui/types.ts`：OverlayType 扩展为 `null | "settings" | "command-palette"`
- `src/app/App.tsx`：注册 ⌘K 快捷键

**工作量估计：** ~200 行新增。

### 4.4 P3：Agent 状态条（Thin Overlay）

**现状问题：**

Agent 的输出和普通 shell 共用同一个 xterm。Claude Code 的 markdown 输出、tool 调用指示、思考过程全是 raw terminal text。用户必须盯着滚动的终端才知道 agent 在干什么——而 agent 的输出速度远快于人类阅读速度。

**设计方案：**

在 agent 运行时，终端顶部浮一个 32px 的状态条：

```
┌──────────────────────────────────────┐
│ 🟡 Claude Code · 运行中 · 已编辑 3 文件 │  ← Agent 状态条
├──────────────────────────────────────┤
│                                      │
│  (xterm 正常渲染 agent 输出)           │
│                                      │
└──────────────────────────────────────┘
```

**信息来源：**
- Agent 名称：`session.agent`
- 运行状态：`session.runState`
- 文件改动计数：`session.changes?.files.length`（通过 gitNonce 定时刷新）

**不做的事：**
- 不解析 agent 的 stdout 来提取结构化事件（这需要为每个 agent 写 parser，维护成本太高）
- 不做完整的 transcript view（这是另一个产品级别的功能）
- 只在有 agent 运行时显示，agent 退出后 3 秒淡出

**实现路径：**
- 在 `TerminalView.tsx` 外层包一个 overlay div
- 信息来自 session store，不需要新的数据源
- 约 60 行新增

**涉及文件：**
- `src/ui/TerminalView.tsx` 或新增 `src/ui/AgentStatusBar.tsx`
- `src/ui/MainArea.tsx`：在 TerminalView 上层渲染 AgentStatusBar

**工作量估计：** ~80 行新增。

## 五、设计品味：核心叙事

Tunara 当前的问题不是做得太少，而是**焦点分散**。它同时在做终端、Git GUI、Agent 检测、设置面板——每一个都做了 70 分，但没有一个做到 90 分。

**有 taste 的做法是砍到只剩核心叙事：**

> "我能同时看到 agent 在做什么，和它改了什么。"

具体对应：

```
左 pane：agent 在跑          右 pane：我在操作
              ↓
      右栏：只读 Review 面板（文件改动 + diff）
```

**优先级排序的逻辑：**

| 优先级 | 功能 | 解决的问题 |
|--------|------|-----------|
| P0 | Split Pane | "我不能同时看 agent 和做自己的事" |
| P0 | DiffPanel → Review（砍 commit UI） | "这个面板又能看又能操作，什么都做不好" |
| P0 | Session 生命周期 | "我不知道 agent 还在不在跑" |
| P1 | 会话持久化 | "关掉 app 重开一切都没了" |
| P1 | 侧栏标题推导优化 | "侧栏标题经常显示错误命令" |
| P2 | Command Palette | "功能越来越多，我找不到入口" |
| P2 | CSS 类提取 | 可维护性，无直接用户感知 |
| P3 | Agent 状态条 | "agent 跑太快我看不过来" |

**每个阶段完成后的交付物：**

- **Phase 1（P0）：** Split Pane + Review 面板 + Session 生命周期 → Tunara 成为“带智能侧栏的真实终端”，而不是 agent 工作台
- **Phase 2（P1）：** 持久化 + 标题优化 → 日常使用不再有割裂感

### 6.1 目录关闭所有会话

**场景：** 用户在一个项目目录下跑了 3-5 个 agent 会话，全部完成后需要逐个关闭——操作繁琐。需要一个按目录批量关闭的入口。

#### 交互设计

在侧栏目录组头部（`DirGroupHeader`）的 `+` 按钮旁加一个 `×` 按钮：

```
┌───────────────────────────────┐
│ 📁 ~/code/project    3  + ×  │  ← × 为关闭全部按钮
│   ✓ Claude Code              │
│   ✓ Codex                    │
│   • 终端                      │
└───────────────────────────────┘
```

**按钮视觉：**
- 位置：紧跟 `+` 按钮右侧，同尺寸（18×18px）
- 样式：与 `+` 按钮一致——`hover-close` CSS 类，`color: var(--c-text-5)`
- 图标：`×`（与 SessionCard 关闭按钮同款 SVG，12×12）
- `title="关闭此目录全部会话"`

**状态流程：**

```
用户点击 ×
  ├─ 组内无 busy 会话 → 直接全部关闭
  └─ 组内有 busy 会话 → 按钮变红 + 提示文字，3 秒后复位
                         ├─ 3 秒内再次点击 → 全部关闭（包括 busy）
                         └─ 超时 → 恢复默认态
```

这与 `SessionCard` 的单个关闭确认机制完全一致（`closeConfirmations` + 3 秒超时 + 二次点击）。

#### Store 层

`src/state/sessions.ts` 新增一个 action：

```typescript
// SessionsState interface 内新增：
closeSessionsInDir: (dir: string) => void;
dirCloseConfirmations: Record<string, number>;  // dir → timestamp
clearDirCloseConfirmation: (dir: string) => void;
```

实现逻辑：

```typescript
closeSessionsInDir: (dir) => {
  const sessionsInDir = get().sessions.filter((s) => s.dir === dir);
  const hasBusy = sessionsInDir.some((s) => isSessionBusy(s));

  if (hasBusy) {
    const lastConfirm = get().dirCloseConfirmations[dir] ?? 0;
    if (Date.now() - lastConfirm > 3_000) {
      // 首次点击：记录确认时间戳，不关闭
      set((state) => ({
        dirCloseConfirmations: { ...state.dirCloseConfirmations, [dir]: Date.now() },
      }));
      return;
    }
    // 3 秒内二次点击：确认关闭
  }

  // 关闭该目录下所有会话
  // 注意：需要处理 split pane 中包含被关闭会话的情况
  for (const s of sessionsInDir) {
    get().closeSession(s.id);
  }

  // 清理确认状态
  set((state) => {
    const { [dir]: _, ...rest } = state.dirCloseConfirmations;
    return { dirCloseConfirmations: rest };
  });
},
```

#### UI 层

`src/ui/Sidebar.tsx` 的 `DirGroupHeader` 组件：

1. 新增 props：`onCloseAll?: () => void`、`confirmClose?: boolean`、`onClearCloseConfirm?: () => void`
2. 在 `+` 按钮右侧渲染 `×` 按钮
3. `confirmClose` 为 true 时按钮 `color` 切换为 `var(--c-error)`
4. 3 秒自动清除确认状态（同 `SessionCard` 的 `useEffect` + `setTimeout` 模式）

#### 约束（硬性）

1. **复用 `closeSession` 逻辑**：不要绕过 `closeSession` 直接 `removeSession`。`closeSession` 内部处理了 split pane 清理、空列表时创建默认终端等边界情况。逐个调用 `closeSession` 即可。
2. **确认机制与 SessionCard 一致**：使用相同的"首次点击记录时间戳 → 3 秒内二次点击确认"模式。不要用 `window.confirm` 弹窗。
3. **Hover 用 CSS 类**：按钮使用 `hover-close` 类（`tokens.css` 已有），不用 JS `onMouseEnter`/`onMouseLeave`。
4. **按钮仅在 `onCloseAll` 存在时渲染**：与 `+` 按钮的 `onNewTerminal` 条件渲染逻辑一致。
5. **关闭顺序**：从后往前关闭（`sessionsInDir.reverse()`），这样如果当前活跃会话在组内，它会最后被关闭，减少中间态的活跃会话跳转。
6. **最后一个会话的处理**：`closeSession` 已有逻辑——`sessions.length === 0` 时自动创建 `~` 默认终端。批量关闭后如果全局无会话，这个兜底会自动触发，不需要额外处理。

#### 涉及文件

| 文件 | 改动 |
|------|------|
| `src/state/sessions.ts` | 新增 `closeSessionsInDir` action + `dirCloseConfirmations` state + `clearDirCloseConfirmation` action |
| `src/ui/Sidebar.tsx` | `DirGroupHeader` 加 `×` 按钮 + 确认态 UI + 从 store 读取/清除确认状态 |

**不涉及的文件：** 不需要改 `SessionCard.tsx`、`types.ts`、`tokens.css`、后端 Rust 代码。

~50 行（store ~25 行 + UI ~25 行）。

新增一个**专用**命令，而不是暴露通用 `shell_run_command`（产品刻意不把通用 shell 执行挂到 `invoke_handler`）：

```rust
// src-tauri/src/modules/editor/mod.rs（新建）
#[tauri::command]
pub fn open_in_editor(editor: String, path: String, line: Option<u32>) -> Result<(), String> {
    // editor 必须命中白名单，参数以独立 arg 传入（绝不拼 shell -c），杜绝注入
    let (bin, goto_style) = match editor.as_str() {
        "vscode"  => ("code",   GotoStyle::Flag),   // code --goto file:line
        "cursor"  => ("cursor", GotoStyle::Flag),
        "zed"     => ("zed",    GotoStyle::Colon),  // zed file:line
        "sublime" => ("subl",   GotoStyle::Colon),
        _ => return Err(format!("unsupported editor: {editor}")),
    };
    // 复用 resolver 找二进制（继承 login shell 的 PATH，与 CLI 检测一致）
    let resolved = crate::modules::resolver::resolve_bin_path(bin)
        .ok_or_else(|| format!("editor not found: {bin}"))?;
    let mut cmd = std::process::Command::new(resolved);
    match goto_style {
        GotoStyle::Flag => { cmd.arg("--goto"); cmd.arg(target(&path, line)); }
        GotoStyle::Colon => { cmd.arg(target(&path, line)); }
    }
    cmd.stdin(Stdio::null()).stdout(Stdio::null()).stderr(Stdio::null());
    cmd.spawn().map_err(|e| e.to_string())?; // detached，不 wait
    Ok(())
}
```

在 `lib.rs` 的 `generate_handler!` 注册 `modules::editor::open_in_editor` 并 `mod editor;`。

前端桥 `src/modules/editor/open.ts`：

```typescript
import { invoke } from "@tauri-apps/api/core";
export function openInEditor(editor: string, path: string, line?: number) {
  return invoke<void>("open_in_editor", { editor, path, line });
}
```

#### 约束（硬性）

1. **编辑器白名单 + 无 shell。** 后端只接受枚举值映射到固定二进制名；`path` / `line` 以**独立 `arg`** 传给 `Command::new(bin)`，绝不走 `sh -lc` 字符串拼接 —— 防命令注入。非法 `editor` 直接 `Err`。
2. **二进制解析走现有 resolver。** 用 `resolver`（已有 `resolve_bin`，登录 shell PATH）找编辑器，未找到返回 `Err`，与 agent CLI 检测同一套路径解析逻辑。
3. **失败要有反馈。** 前端 catch 到 `Err` 时用 `addToast`（error variant）提示「未找到编辑器：{name}」，复用现有 Toast，不要静默吞掉。
4. **按钮 `stopPropagation`。** `↗` 的 `onClick` 必须 `e.stopPropagation()`，否则会连带触发 `toggleFile` 展开/收起 diff。
5. **v1 文件级打开（行号可选）。** 文件行按钮传 `line` 省略（编辑器打开到文件顶部）。当前 `DiffPanel` 未逐行解析 hunk 起始行，行级跳转留到「diff 行内点击跳转」时再做，避免引入 hunk 解析复杂度。
6. **路径拼接在前端完成且规范化。** 传给后端的是绝对路径（`repoPath + "/" + file.path`），后端不做 cwd 推断。
7. **悬停显形用 CSS**（`.diff-file-row:hover` 控制 `↗` 的 opacity），按钮本身复用 `hover-bg`；不要用 JS hover state，保持与 DiffPanel 其余按钮一致。
8. **`externalEditor` 进 `PERSIST_KEYS`**，默认 `"vscode"`；`loadSettings` 里对非法值做白名单校验（与 `terminalTheme` 的校验写法一致）。

### 6.4 Command Palette 使用权重排序（P2）

**场景：** `CommandPalette.tsx` 的命令列表是**静态顺序**（会话切换 → 新建终端 → 切换侧栏 → … → 设置）。常用命令（如频繁切到某个会话、反复分栏）每次都要从头找。让最近用过的命令在空查询时浮到前面，是 Raycast / VS Code Palette 的标配体验。

#### 前端设计

不改 Palette 的视觉结构（输入框 + 分组列表 + 选中高亮都保留），只改**排序**：

- **空查询：** 每个分区（「会话」「操作」）内，按「最近使用时间」降序排，未用过的按原始插入顺序兜底
- **有查询：** 主排序仍是匹配质量（匹配位置 `indexOf` 越靠前越优先），最近使用时间作 tiebreaker
- 选中态、上下键导航、`scrollIntoView` 逻辑全部不变（`selectedIndex` 已在 query 变化时归零）

#### Store 层

`ui.ts` 记录命令使用时间戳：

```typescript
// UIState 内新增：
commandUsage: Record<string, number>;   // commandId → 最近使用时间戳
recordCommandUse: (id: string) => void;

recordCommandUse: (id) =>
  set((s) => {
    const next = { ...s.commandUsage, [id]: Date.now() };
    // 上限保护：只保留最近 50 条，防止无界增长
    const entries = Object.entries(next).sort((a, b) => b[1] - a[1]).slice(0, 50);
    return { commandUsage: Object.fromEntries(entries) };
  }),
```

#### UI 层

`src/ui/overlays/CommandPalette.tsx`：

1. 构建 `commands` 时给每个 cmd 带上 `originalIndex`（数组下标），作为稳定排序兜底
2. 每个 `cmd.action()` 执行前调用 `useUIStore.getState().recordCommandUse(cmd.id)`（在 `onClose` 之前）
3. `filtered` 之后追加排序：
   ```typescript
   const usage = useUIStore((s) => s.commandUsage);
   const ranked = [...filtered].sort((a, b) => {
     if (q) {
       const ia = a.label.toLowerCase().indexOf(q);
       const ib = b.label.toLowerCase().indexOf(q);
       if (ia !== ib) return ia - ib;
     }
     const ua = usage[a.id] ?? 0;
     const ub = usage[b.id] ?? 0;
     if (ua !== ub) return ub - ua;
     return a.originalIndex - b.originalIndex;
   });
   ```
4. 分区分组（`sections` Map）与渲染基于 `ranked` 而非 `filtered`

#### 约束（硬性）

1. **`commandUsage` 持久化且有上限。** 进 `PERSIST_KEYS`（或单独 localStorage key），`recordCommandUse` 内截断到最近 50 条，避免会话切换 id（`switch-<sessionId>`）无限堆积。
2. **排序必须稳定。** ties 一律由 `originalIndex` 兜底，避免列表抖动；已删除会话的 `switch-*` 用量自然不再出现在 `commands` 里（id 不匹配即忽略），无需主动清理。
3. **不改 Palette 视觉与键盘交互。** 仅替换数据源排序；选中条、`↑/↓`、`Enter`、`scrollIntoView`、分区标题均不动。
4. **不引入模糊匹配库。** 继续用 `includes` / `indexOf`，纯本地 `sort`，零新依赖。
5. **记录时机在 action 内、onClose 之前**，键用 `cmd.id`（会话切换按具体 sessionId 计权）。

#### 涉及文件

| 文件 | 改动 |
|------|------|
| `src/state/ui.ts` | `commandUsage` + `recordCommandUse` + 持久化 |
| `src/ui/overlays/CommandPalette.tsx` | `originalIndex`、排序、action 内打点 |

#### 工作量

~50 行（store ~15 行 + UI ~35 行）。

### 6.5 Linux 支持收尾（P2）

> 类型：平台工程，非纯 UI。`tauri.conf.json` 已含 `linux` 的 deb/rpm/appimage 打包配置，`tauri.linux.conf.json` 已设 `decorations:false` + `transparent:true`。剩余工作是把 macOS 专有的窗口与材质逻辑做成跨平台收尾。

#### 待办清单

1. **窗口控制按钮（前端）。** Linux 走 `decorations:false`，没有原生最小化/最大化/关闭。`Titlebar.tsx` 需要在非 macOS 平台渲染一组自绘窗口控制（min/max/close），调用 `getCurrentWindow().minimize()/toggleMaximize()/close()`。平台判定用已注册的 `tauri-plugin-os`（或 `navigator.platform`）。macOS 继续用 `titleBarStyle:"Overlay"` 的红绿灯，不出自绘按钮。
2. **毛玻璃降级。** `apply_vibrancy` 已 `#[cfg(target_os = "macos")]` 门控（`lib.rs`），Linux 不会调用。需确认 `--c-bg-2-glass` 等依赖 `backdrop-filter` 的 token 在 Linux WebKitGTK 下有可接受的回退（必要时为非 macOS 提供不透明背景变量）。
3. **字体。** 确认 JetBrains Mono 在 Linux 包内可加载（`document.fonts.load`），缺字时回退到 `monospace`。
4. **trafficLightPosition / Info.plist** 等 macOS-only 配置确保不影响 Linux 构建。
5. **CI 增加 Linux target 构建**（appimage + deb），冒烟测试 PTY / OSC / git 读路径（这些走 `$SHELL -lc`，本就跨平台）。

#### 约束（硬性）

1. **单一代码库 + 平台分支。** 所有 macOS-only 原生 API 必须 `#[cfg]` 门控（Rust）或运行时平台判定（前端），不开 Linux 专用 React 分叉。
2. **窗口控制只在缺原生装饰时出现。** macOS（Overlay + 红绿灯）不渲染自绘按钮，避免双份控制。
3. **材质有回退。** 任何 `backdrop-filter` 依赖处都要有不透明 fallback，防止 Linux 下面板与背景糊成一片（见 §五 Gotchas：相邻嵌套面板必须有可见区分）。

#### 涉及文件

`src/ui/Titlebar.tsx`（平台感知窗口控制）、`src/styles/*`（材质回退变量）、`src-tauri/src/lib.rs`（确认 cfg 门控）、CI 配置。

#### 工作量

中等，主要是 Titlebar 自绘控制（~100 行）+ 跨平台验证。

### 6.6 Homebrew 分发（P2）

> 类型：发布工程，非 UI。目标：`brew install --cask tunara`。

#### 方案

1. **构建产物。** `tauri build` 产出签名 + 公证的 `.dmg`（已有 `updater` 指向 GitHub Releases，artifact 路径现成）。
2. **自建 tap。** 先建 `homebrew-tunara` tap 仓库，提供 cask：
   ```ruby
   cask "tunara" do
     version "1.0.2"
     sha256 "<dmg-sha256>"
     url "https://github.com/.../releases/download/v#{version}/Tunara_#{version}.dmg"
     name "Tunara"
     desc "Terminal with an intelligent session sidebar"
     homepage "https://github.com/..."
     app "Tunara.app"
     auto_updates true   # 应用内 updater 已接管升级
   end
   ```
3. **发布 CI 自动更新 cask。** 每次 release 后用 CI 计算 `.dmg` 的 sha256，回写 cask 的 `version` + `sha256` + `url`。

#### 约束（硬性）

1. **必须签名 + 公证。** 未公证的 cask 会触发 Gatekeeper 警告，体验极差 —— 公证是分发前置条件。
2. **cask 版本/sha256 与 Release artifact 强绑定**，由 CI 自动同步，杜绝手填漂移。
3. **先走自建 tap**；提交 homebrew-cask 主仓有「知名度」门槛，早期不强求。
4. **`auto_updates true`** 与应用内 `tauri_plugin_updater` 一致，避免 brew 与内置 updater 双重升级打架。

#### 涉及文件

`homebrew-tunara`（新仓库 / tap）、发布 CI 配置、`tauri.conf.json`（签名 + 公证配置确认）。

#### 工作量

发布管线 ~半天，cask 本体 ~30 行。

## 七、不做清单

以下功能明确不在规划范围内，记录理由避免后续重新讨论（原 §六）：

| 不做的事 | 理由 |
|----------|------|
| Agent stdout 结构化解析 | 需要为每个 agent 写 parser（至少 11 种），维护成本远超收益 |
| 内置 Agent SDK 调用 | 决策 D3 已确定走 CLI 而非 SDK，不变 |
| 完整的 terminal multiplexer（tmux 级 tile） | 递归分栏的复杂度不值得，一次 split 覆盖 95% 场景 |
| 内置文件编辑器 | Tunara 不是 IDE，文件编辑交给 agent 或外部编辑器 |
| 远程 SSH 连接 | 目标用户在本机跑 agent，不是运维场景 |
| Tab 拖拽排序 | 当前 tab 数量不大（通常 <8），ROI 不高 |
| 插件系统 | 产品太早期，抽象层会变成负担 |
