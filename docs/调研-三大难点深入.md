# Conduit 三大难点深入调研：会话编排 · Git 集成 · AI Agent 接入

> 调研日期：2026-06-18
> 承接《调研-Conduit终端方案.md》第三部分。聚焦三个高风险/高不确定模块，外加 Warp 产品对标。
>
> **核心结论（先看）**：
> 1. **AI Agent 接入不确定性已大幅消除**——目标 agent 为 **Claude Code（CC）+ Codex（CX）两家**（**已砍掉 Cursor，见决策 D2**），二者均确认提供 **headless + JSONL 事件流 + 会话续接（resume）**，能力对称、形态高度一致。这把"最大风险点（⭐⭐⭐⭐）"降到了 **⭐⭐⭐ 可控工程**。
> 2. **会话编排 ≈ AI 接入的同一件事**：一个会话 = 一个 agent 子进程，状态机（running/fresh/done）由解析其事件流直接驱动。三块里这是连接器。
> 3. **Git 集成**选型明确：**`git2`（libgit2 绑定）只负责读**（status/diff/ahead-behind）；**所有写操作（commit + push）走系统 `git` CLI**（决策 D4），借用户已配好的凭证/hooks/签名，避开 git2 credential 回调与手写 commit 树的坑。保持 ⭐⭐⭐。
> 4. **Warp 的 Oz 编排器**为 Conduit 的"统一面板编排多 agent"产品方向提供了**外部验证**（非技术正确性证明），可作方向性对标。

## 难点一：AI Agent 接入（原评级 ⭐⭐⭐⭐ → 实测 ⭐⭐⭐）

> **范围变更（D2）**：原设计稿含 CC/CX/**CU(Cursor)** 三家；**本项目砍掉 CU，只接 CC + CX**。原因：① 二者能力对称（都有 headless+JSONL+resume），统一抽象更干净；② 规避 CU headless 的已知坑（不加 `--force` 假死、文档/版本对 resume 说法不一需逐版本实测）；③ 砍掉后无需为单一异类做特殊分支。设计稿里 CU 的角标/选项在 UI 移除（或留位灰置"暂不支持"）。

### 1.0 关键事实：两家 agent 形态对称

CC 与 CX 均支持「非交互 headless + 结构化事件流 + 会话续接」，用**同一套抽象**接入：

| 能力 | Claude Code (CC) | Codex (CX) |
|---|---|---|
| headless 命令 | `claude -p "<prompt>"` | `codex exec "<prompt>"` |
| 流式事件 | `--output-format stream-json --verbose` | `--json`（JSONL） |
| 结构化结果 | `--output-format json`（`result` 字段）/ `--json-schema` | `--output-schema <path>` / `-o <file>` |
| 会话续接 | `--resume <session_id>` / `--continue` | `resume <thread_id>` / `resume --last` |
| 自动批准/权限（D1 直接写盘） | `--permission-mode acceptEdits` | `--sandbox workspace-write` |
| 启动加速 | `--bare`（跳过 hooks/MCP/CLAUDE.md 发现） | `--ignore-user-config` |

> 这正是 Warp Oz 做的事（§4）：把异构 agent 统一成「输入 prompt → 流式事件 → 最终结果」的 harness 抽象。Conduit 只需做单机版。
>
> ⚠️ **schema 实测前提**：下文各事件字段来自官方文档 + 三方 cheatsheet，**需在钉定的 CLI 版本上实测校正**（见 §1.5 版本矩阵）。parser 须对缺字段容错（返回 None 而非 panic）。

### 1.1 Claude Code（CC）事件流详解

```bash
claude -p "修复鉴权测试" \
  --output-format stream-json --verbose --include-partial-messages \
  --allowedTools "Bash,Read,Edit" \
  --resume "<session_id>"        # 可选，多轮
```
事件类型（逐行 JSON）：
- `system/init`：**首个**事件，含 model、tools、mcp、plugins、`session_id`。→ 拿 session_id 存起来供 resume。
- `stream_event`：增量 token，文本在 `.event.delta.text`（`delta.type=="text_delta"`）→ 喂给内联 AI 块做打字机效果。
- `assistant`：完整 assistant 消息（含 tool_use）。
- `system/api_retry`：重试事件（attempt/max_retries/error 分类）→ 可在 UI 显示"重试中"。
- `result`：**完成标志**，字段：
  ```json
  {"type":"result","subtype":"success",   // 或 error 类
   "session_id":"...","is_error":false,
   "result":"最终答复字符串",
   "num_turns":2,"duration_ms":12345,"total_cost_usd":0.0123,"usage":{...}}
  ```
  → `subtype=="success" && !is_error` ⇒ 状态机置 `fresh`；`is_error` ⇒ 失败通知。

> ⚠️ **已知坑**：社区有偶发 `result` 事件缺失 / 发完后进程 hang 的 issue（#1920 #8126 #25629）。**对策（统一生命周期，缺一不可）**：
> - ① **进程 exit 兜底**：监听子进程 `exit`，stdout EOF 后 await 真实退出码补发 Done/Failed。
> - ② **wall-clock + idle timeout**：`exit` 兜底的前提是 stdout 会 EOF；但最坏的 hang 恰恰是 **stdout 不 EOF、进程不退出**，此时只靠 ① 会永久 running。必须再加：总时长上限（wall-clock）+ 无新事件静默上限（idle）。任一超时即 **kill 子进程 → 发 `Failed{timeout}` → 释放并发名额 → 持久化失败原因**。
> - ③ **手动 cancel**：用户可随时取消，走与超时同一条 kill + 清理路径。
> 三者收敛到同一个"终止 → 清理"出口，保证任何分支下会话都不会卡在 running。退出码也要读，进失败诊断。

### 1.2 Codex（CX）事件流详解

```bash
codex exec "重构 webhook" --json \
  --sandbox workspace-write \
  -o /tmp/last.txt              # 最终消息落文件
# 续接：codex exec resume <thread_id> "<new>"  或  resume --last
```
事件类型（JSONL）：
- `thread.started`：含 `thread_id` → **= 会话 id，存起来供 resume**。
- `turn.started` / `turn.completed`（含 `usage`）/ `turn.failed`。
- `item.started` / `item.completed`：操作粒度——`command_execution`、`agent_message`、`file_change`、`mcp_tool_call`。
- **完成判定**：`turn.completed`；最终答复在最后一个 `agent_message` item。
- 退出码：有凭证时 `codex login status` 返回 0，可用于预检。

### 1.3 接入架构（统一 Harness 抽象）

```
trait AgentHarness {
  fn spawn(prompt, cwd, session_id?) -> ChildProcess(stdout=JSONL)
  fn parse_event(line) -> AgentEvent   // 归一化：Started{session_id} | Delta{text}
                                        //         | ToolUse{..} | FileChange{..}
                                        //         | Done{ok, result, cost?} | Failed{err}
}
impl: ClaudeHarness / CodexHarness     // 各自封装 §1.1–1.2 的 flags 与解析（CU 已移除）
```
- **每个会话 = 一个 agent 子进程**。Rust 端用 `tokio::process` spawn，stdout 逐行解析为归一化 `AgentEvent`，经 Tauri `Channel` 推前端。
- 归一化事件直接驱动：① 内联 AI 块（Delta 打字机 + ToolUse 折叠）；② 状态机（§2）；③ diff 面板（FileChange → 触发 git diff 刷新）。
- ⚠️ **渲染模型必须先拍板（二轮 P0，跨难点一/二的实现约束）**：设计稿要求"AI 回复内联在终端流里"，但 **xterm.js 有自己的 buffer/scrollback/selection，React 组件不能像普通 DOM 那样插进 xterm 输出流**。这不是一个 `InlineAgentBlock.tsx` 组件能解决的，而是先要选定 transcript 架构。本项目的取舍（详见实施文档 §4.4）：
  - **MVP**：agent 会话用**独立的 React transcript 视图**（不伪装成 xterm），shell 会话仍用 terax 的 xterm 接线——保住"终端核心白送"。两类会话由 `kind` 分流渲染。
  - **目标形态（二期/风险项）**：外层 React 虚拟列表承载 shell block + agent block，xterm 只负责当前交互块。这最贴设计稿"终端流内联 AI"，但**代价高**：shell 历史要 block 化（靠 OSC 133 切命令边界）、跨 block 的滚动/选择/复制都要自实现，xterm 接线**不能再直接照搬 terax**。故列为 MVP 之后的演进，不进 MVP 范围。
- **"应用补丁"语义（D1，已定：直接写盘）**：agent 用 `--permission-mode acceptEdits`(CC) / `--sandbox workspace-write`(CX) 直接改文件，UI 只展示结果 + diff。
  - ⚠️ **语义变化**：这把交互从"AI 提议→你批准→应用"变成"AI 已改→你事后看 diff→决定提交/丢弃"。设计稿的「应用补丁」按钮**降级为"已应用 ✓"提示或移除**，并应提供「丢弃改动」后悔药。
  - 🔒 **写盘护栏（P1-4 + 二轮 P0/P1 升级）**：
    - ① agent 只能在选定 **cwd** 下运行（`current_dir`）。
    - ② **baseline snapshot 升为 D1 必备子决策（二轮 P0）**：D1 直接写盘后，最核心的风险不是"展示 diff"，而是**区分 agent 本轮改动与用户启动前已有的 dirty work**。必须在 spawn 前记录 baseline（HEAD oid + index + worktree 指纹），完成后据此计算**本轮新增 diff**；「丢弃改动」**只回滚本轮改动**，对启动前已存在的改动绝不动手。无法安全区分时（如 baseline 期间外部又改了文件）**拒绝自动丢弃并提示人工处理**，不赌。
    - ③ **敏感/受保护路径优先阻断（二轮 P1，修正措辞）**：原写"敏感写入给持久通知"不准确——真实 agent 的 sandbox/permission 通常在写**受保护路径**（`.git/`、`~/.ssh`、CI 配置等）时**直接 fail 或拒绝**，而非"写成功后通知"。应改为：**受保护路径优先阻断/失败提示；只有实际发生的高风险改动（如跨目录写入、批量删除）才给持久通知**。protected-path 须有 fixture 覆盖（见 §1.5）。

### 1.4 风险评估
- **不确定性主因已消除**：CC/CX 都有稳定 headless + 事件流 + resume，不需逆向或 hack。
- **剩余风险**：CC 的 result-hang issue（用进程 exit 兜底）、CLI 版本/字段漂移（见 §1.5）。
- **重估难度：⭐⭐⭐**（工程量在"统一抽象 + 两套解析 + 流式 UI"，无研究性未知）。

### 1.5 版本矩阵 + golden JSONL fixtures（P1-5，落地必做）
解析器是脆弱点（字段名随 CLI 版本漂移）。对策：**钉死版本 + 为每家存样例 JSONL 做回归测试**。

| Agent | 本次核验版本 | fixtures（每家至少 3 条） |
|---|---|---|
| Claude Code | `2.1.178` | ① 成功 run（含 `system/init`+`result success`）② 失败 run（`is_error`/无 result）③ 改文件 run |
| Codex | `0.141.0` | ① `thread.started`+`turn.completed` ② `turn.failed` ③ 含 `item.completed/file_change` |

- fixtures 存 `src-tauri/tests/fixtures/{cc,cx}/*.jsonl`，`parse_line` 写单测吃这些样例断言归一化输出。
- 升级 CLI 时先重录 fixtures、跑测，再改 parser——把"版本漂移"变成可见的测试失败。
- **fixtures 还须覆盖（二轮 P1）**：CC 的 `assistant.tool_use`、编辑文件、错误事件；CX 的 `turn.failed`（断言提取出**原始错误字段**而非硬编码字符串）；以及 **protected-path 写入被阻断**的事件（验证 §1.3 ③）。

> ⚠️ **版本矩阵只钉了 parser，没钉启动环境（二轮 P1）**：fixtures 保证"事件解析对"，但保证不了"CLI 能被找到"。**macOS GUI app（Finder 双击 `.app`）的 PATH 与 login shell 不同**——`claude`/`codex`/`git` 在 `cargo tauri dev`（继承终端 PATH）下能跑，打包后双击启动可能全部 `not found`。这不是"CLI 未装"，是 app bundle 环境问题。须把**"打包后双击 .app 的 CLI 发现 + 登录态预检"纳入 Agent 接入风险与验收**（对策见实施文档 §3.3.4：从 login shell 解析 PATH / `fix-path-env` + 设置里可覆盖绝对路径）。

## 难点二：会话编排 + 状态机（⭐⭐⭐，与难点一强耦合）

### 2.1 本质
设计稿的会话挂着 agent，所以「会话编排」= **agent 进程生命周期管理 + 由事件流驱动的状态机 + 分组/持久化**。不是独立难点，是难点一的上层。

> ⚠️ **渲染层约束（二轮 P0，与难点一 §1.3 同一问题）**：会话编排要在主区把 shell 会话与 agent 会话一起呈现，但 **xterm buffer 与 React DOM 是两套模型，不能天然混排**。因此编排层不能假设"AI 块直接插进 xterm"，必须先选定 transcript 架构（MVP=agent 独立 transcript 视图 / 目标=外层 block 虚拟列表），见 §1.3。本项目把它当作难点一/二的**交叉实现约束**处理，不单列为"难点四"——它无研究性未知，只是要求实现前定渲染模型。

### 2.2 状态机（对应设计稿三态）

```
            spawn agent 子进程
   (new) ───────────────────────▶ running
                                    │  ├─ Delta/ToolUse 事件 → 更新内联块 + 进度条(可用 num_turns/已知步数估算)
                                    │  ├─ Done{ok:true}  ──▶ fresh   (绿勾"刚完成" + 标题绿点)
                                    │  ├─ Done{ok:false} ──▶ failed  (红，进通知中心，持久)
                                    │  └─ 进程 exit(兜底) ──▶ done   (灰点 "exit N")
   fresh ──(用户切走/超时 N 秒)──▶ done
```
- **进度条**：设计稿 running 卡有 3px 进度条。真实进度无法精确——用启发式：已完成 tool_use 数 / 预估、或 `num_turns` 增长、或纯"不确定脉冲"动画。建议 MVP 用不确定动画，别假装精确。
- **运行中呼吸点 `pulseDot`**：纯 CSS，状态=running 时挂上。

### 2.3 会话数据模型（落地版，扩展自设计稿 §1.5）

```ts
type Session = {
  id: string;
  kind: 'shell' | 'agent';     // 统一容纳"新建终端"与"新建 Agent"两类
  agent?: 'CC' | 'CX';         // 仅 agent 会话（CU 已移除）
  agentSessionId?: string;     // CC=session_id / CX=thread_id（多轮 resume 用）
  title: string;               // 由首条 prompt 截取或 agent 命名
  dir: string; branch: string; // cwd + 当前 git 分支
  status: 'running' | 'fresh' | 'failed' | 'done';
  procId?: number;             // 后端 agent_spawn / pty_open 返回的 id
  // ⚠️ 内联流不要存成 AgentEvent[]（长会话几千条 delta → 每事件全量复制数组 = O(n²)）。
  //    delta 文本累积进字符串 buffer；只把"块边界"（toolUse/fileChange/轮次）存成少量结构。
  //    ⚠️ 但"字符串累积"不等于零成本（二轮 P2）：高频 reply+=delta 后整串进 React state，
  //       触发的 re-render + diff 才是真成本（不是字符串拼接本身）。须在 UI 层节流：
  //       后台 chunk buffer 收流，requestAnimationFrame 或 50–100ms 合并提交到 state。
  reply: string;               // 累积的 AI 正文（打字机）
  blocks: AgentBlock[];        // 少量：toolUse / fileChange / 多轮分隔
  // diff 面板（来自 git，难点三）：
  changes?: { files: FileChange[]; summary: string; commitSuggest?: string };
};
// 分组：按 dir 归组（~/orbit / ~/web / ~/infra），= 设计稿侧边栏分组
```

### 2.4 持久化
- **必须持久化的终态（P1-6，MVP 必做）**：每个会话的 `status / result / error / updatedAt` 用 `tauri-plugin-store` 落盘。否则应用重启后会**把已结束会话错显为 running，或丢失失败原因**。这是低成本、高收益的修正，**不可推到二期**。
- **会话元数据**：列表/分组/`agentSessionId`/`dir`/`branch` 一并存（store JSON）。
- **transcript 重放（二期，D5）**：切走的会话切回看完整历史对话需保活进程 + 内存 `reply`/`blocks`；**重启后完整重放**（含逐条对话）放二期（需 resume + 重放）。MVP 重启后会话显示终态摘要即可，不恢复逐字历史。
- **真实终端 vs agent 会话**：两类会话由 `kind: 'shell'|'agent'` 区分；shell 类的 PTY 难点已由 terax 解决。

### 2.5 风险
- 状态机本身简单；风险在**事件流的边界情况**（agent 崩溃/超时/result 缺失/永久 hang）。覆盖手段是 §1.1 的**统一生命周期**：exit 兜底 + wall-clock/idle timeout + 手动 cancel 收敛到同一清理出口；任何分支都 kill 子进程、补终态事件、释放并发名额、持久化失败原因，**杜绝会话永久 running**。
- 多会话并发 N 个 agent 子进程 → 资源/限流（别让用户开 20 个 CC 跑爆机器，需软上限或排队）。**并发名额的获取与释放必须原子**：超时/取消/正常完成三条路径都要确保名额归还，否则名额泄漏会让后续 spawn 被误拒（实施细节见实施文档 §3.3.3）。

## 难点三：Git 集成（⭐⭐⭐，terax 无，需自建）

### 3.1 选型结论：用 `git2`（libgit2 绑定）

| 方案 | 取舍 |
|---|---|
| **`git2`（libgit2-rs）** ✅ 推荐 | 成熟稳定、线程安全、读写全支持、不依赖系统装没装 git、无需起子进程解析文本 |
| `gitoxide`(`gix`) | 纯 Rust、快，但仍在成熟中，写操作/边角不如 libgit2 全 |
| 调 `git` CLI | 最简单上手，但依赖用户环境有 git、要解析非结构化文本输出、跨平台引号坑 |

设计稿 diff 面板需要的能力全在 `git2` 覆盖范围内。

### 3.2 diff 面板需要的数据 → git2 API 映射

设计稿右栏（§《方案》1.3）需要：文件列表(M/A 标记 + 路径 + `+x −y`)、首个文件 mini diff(hunk/增删行)、`领先 X 落后 Y`、commit 输入 + commit&push。

| 设计稿元素 | git2 实现 |
|---|---|
| 文件状态列表 + M/A 标记 | `Repository::statuses()` → `Status::WT_MODIFIED/INDEX_NEW/...` |
| 每文件 `+x −y` 行数 | `Diff::stats()` / 逐 `DiffDelta` 累计 insertions/deletions |
| mini diff（hunk/增删行） | `repo.diff_index_to_workdir()` / `diff_tree_to_workdir()` → 遍历 `Diff::foreach`(hunk/line 回调)，line origin `+`/`-`/` ` |
| 摘要 `3 文件 · +26 −6` | 聚合上面 stats |
| 当前分支 ⎇ | `repo.head()?.shorthand()` |
| `origin/<branch> · 领先 0 · 落后 0` | `repo.graph_ahead_behind(local_oid, upstream_oid)`；**但须结构化降级（二轮 P1）**：无 upstream / detached HEAD / unborn branch 都不该返回 `Err` 连累整个面板，返回 `{ upstream?, ahead?, behind?, state: ok\|noUpstream\|detached\|unborn }`，UI 据 state 显示而非报错。**ahead/behind 不是 diff 面板的硬依赖**——文件列表/diff 不能因 upstream 缺失而不可用。 |
| 提交 | **走 CLI**：`git commit -m <msg>`（D4，下详） |
| **提交并推送** | **走 CLI**：`git push`（D4，下详） |

> **读用 git2、写用 CLI 的分工（D4 已定）**：git2 真正强于 CLI 的地方是**读的结构化输出**（status/diff/ahead-behind 直接拿到字段，不必解析文本）。而写操作用 git2 要手写 `add_all`+`Signature`+tree+parent 一大段易错代码，且 push 还要自己实现 credential 回调。**所以 commit + push 全走系统 `git`**——一行命令，自动继承用户 `user.name/email`、commit hooks、GPG 签名、credential helper / ssh-agent。

### 3.3 写操作（commit + push）走 CLI 的关键护栏（P2-7）
`git` CLI 在自动化里**可能阻塞等交互输入**（要密码/passphrase 的 remote）。必须：
- **禁交互提示**：`GIT_TERMINAL_PROMPT=0`（+ 视情况 `GIT_ASKPASS`），让需要凭证时**立即失败而非挂起**。
- **设超时并真正 kill（二轮 P1，强化）**：仅 `timeout(Command::output())` **不等于**超时 kill——future 被丢弃后底层 `git`/`ssh` 进程可能继续存活，留下残留进程。实现约束写死为 **`spawn()` + `kill_on_drop(true)` + `tokio::select!`**：超时分支显式 `kill().await` + `wait()` 后再返回错误（实施细节见实施文档 §3.4.2）。
- **捕获 stdout/stderr**：失败时把 stderr（如 "Authentication failed" / "could not read Username"）**回显到 UI**，提示"请先在终端完成登录/凭证配置"。
- commit 同理捕获错误（如 pre-commit hook 失败、无改动可提交）。

> diff 面板的 commit message 还可让 agent 生成（设计稿 `commitSuggest` 字段，§2.3）——CC/CX 跑一条 `生成 commit message` 即可。

### 3.4 性能
- status/diff 在大仓可能慢 → 别在每次按键刷新；用文件系统 watch（`notify` crate）或 agent `FileChange` 事件**去抖触发**刷新。
- **单文件超大 / 二进制 diff 也是性能源（二轮 P2）**：性能不只来自刷新频率。`git_diff` 把 patch 转 UTF-8，二进制会丢内容、超大/minified 文件会卡 UI；300px mini diff 不该加载无限 patch。须加 **单文件 diff 行数/字节上限 + binary 标记 + 按需展开加载 + 超限提示**（实施细节见实施文档 §3.4.1 / §4.5）。

### 3.5 风险
- push/commit 走 CLI 的阻塞与认证（见 §3.3：禁交互 + 超时 + **超时须真 kill 子进程** + stderr 回显）。
- 大仓 diff 性能（去抖 + 后台线程 + **单文件 diff 上限/binary 标记/按需展开**，见 §3.4）。
- 常见仓库状态降级：无 upstream / detached / unborn 时 ahead-behind 结构化返回，不连累 diff 面板（见 §3.2）。
- 难度 ⭐⭐⭐ 维持，无未知研究项。

## 第四部分：Warp 产品对标（用户要求）

### 4.1 Warp 现状（2026-04 起）
Warp 已从"终端"转型为 **Agentic Development Environment**：Rust + GPU，源码已公开（双授权 **MIT / AGPL v3**），核心是 **Oz** 云编排器。

### 4.2 Oz 编排——对 Conduit 产品方向的外部验证
> "Companies can ask Oz to spin up cloud agents that directly use **Claude Code, Codex, and Warp Agent**… launch, track, govern, and steer any of these agents in one unified control plane."
> "父 run 分析任务拆子任务，**派 Claude Code 子 agent 做需精审的步骤、派 Codex 子 agent 做高频编辑**，汇总结果解决冲突，返回单一产出（PR/报告/Slack 回复）。"

**这表明 Conduit 的产品方向有外部验证**（Warp 在做类似事——统一面板编排异构 agent），但**仅是方向性背书，不构成对 Conduit 技术选型正确性的证明**。可借鉴的点：
1. "侧边栏管多个 agent 会话 + 终端内联 + diff/PR" 是有人在做的方向。
2. Warp 走 local（实时人在环）+ cloud（自治后台）双模式；Conduit MVP 做 **local 单机版**即可对齐其 local 模式。
3. Warp 的 task 产生"持久记录（status/metadata/transcript/outputs）"——与难点二 §2.4 的终态持久化设计思路一致（互为参照，非因果）。

### 4.3 Conduit 与 Warp 的差异定位
- Warp：自研 GPU UI + 云基础设施 + 团队治理，重。
- Conduit：Tauri + xterm.js（Web UI，颜值靠 CSS）+ 本地 agent 子进程，轻；**护城河在 UI 质感 + 本地多 agent 编排体验**，不与 Warp 的云基础设施正面竞争。
- 参考但不照抄：Warp 的 agent 模式交互（计划/检查点/逐步批准）值得借鉴进 Conduit 的内联块。

## 汇总：三大难点重估

| 难点 | 原评级 | 实测评级 | 关键依据 | 残留风险 & 对策 |
|---|---|---|---|---|
| AI Agent 接入 | ⭐⭐⭐⭐ | **⭐⭐⭐** | CC+CX 均有 headless+JSONL+resume，形态对称（CU 已砍） | CC result-hang（exit 兜底）/ 版本漂移（钉版本+golden fixtures+解析容错） |
| 会话编排+状态机 | ⭐⭐⭐ | ⭐⭐⭐ | = agent 进程生命周期+事件驱动状态机，可复用 terax 进程管理 + tauri-plugin-store | 事件边界(超时/崩溃)、终态必持久化、并发软上限、delta 用字符串 buffer 防 O(n²) |
| Git 集成 | ⭐⭐⭐ | ⭐⭐⭐ | git2 读全覆盖 diff 面板所需；写走 CLI | CLI push/commit 阻塞（禁交互+超时+stderr 回显）、大仓性能（去抖） |

**总结论**：原本最吓人的 AI 接入因三家 headless 标准化而**从研究性风险降为常规工程**；三块的真实工作量集中在「**一套 Harness 抽象 + 三套事件解析 + 流式内联 UI + git2 读 + 状态同步**」，**无阻断性未知**。Warp/Oz 的存在进一步证明架构方向正确。
