# Conduit 产品评估：竞品分析与演进方向

> 基于 M0-M3 + P0-P3 全部落地后的项目全量审读、2026 年中竞品调研，给出现状评估、差异化判断和下一阶段功能建议。

---

## 一、项目现状评估

### 1.1 已完成能力

Conduit 代码量约 4500 行（~2500 行 TypeScript + ~2000 行 Rust），M0-M3 + 产品规划文档中的 P0-P3 功能已基本全部落地：

| 里程碑 | 状态 | 核心交付 |
|--------|------|---------|
| M0 真实 Store | ✅ | Zustand 双 store（sessions + ui），Tauri Store 持久化 |
| M1 多会话 | ✅ | 多 PTY 会话、侧栏按目录分组、tab 导航 |
| M2 Agent 接入 | ✅ | 11 种 AI agent CLI 自动检测（CC/CX/AM/GM/CP/CR/DR/OC/PI/AG/DV） |
| M3 Git Diff 面板 | ✅ | git2 读操作、DiffPanel 只读审查、文件展开 diff |
| P0 Split Pane | ✅ | 水平/垂直分栏、拖拽分栏线、⌘D/⌘⇧D 快捷键 |
| P0 Session 生命周期 | ✅ | runState（idle/running/done/failed）、脉动动画、状态点 |
| P1 会话持久化 | ✅ | Session 列表 + UI 布局跨重启恢复 |
| P1 侧栏标题推导 | ✅ | OSC 133 shell integration 命令/agent 检测 |
| P2 Command Palette | ✅ | ⌘K 触发、会话切换、操作入口、fuzzy search |
| P3 Agent 状态条 | ✅ | 终端顶部浮条、agent 名称 + 运行状态 + 文件改动计数 |

### 1.2 代码质量

**架构判断：清晰、分层合理、无明显技术债。**

- 前端分层明确：`src/ui/`（Conduit 壳层）vs `src/modules/`（terax 复用层）vs `src/state/`（Zustand store）
- 后端 Rust 模块职责边界分明：`resolver/`（CLI 路径发现）、`process/`（子进程管理）、`git/`（结构化读取）、`agent/`（preflight 检测）、`pty/`（终端管理）
- 样式系统一致：CSS 变量在 `tokens.css` 精确映射设计稿 token，hover 效果统一用 CSS 类
- 状态管理规范：Session 存事实字段，UI 组件通过 derive 函数获取展示态

**值得注意的技术点：**

- `CommandResolver` 三层优先级解析（用户覆盖 → login shell PATH → 系统路径），解决了 macOS GUI app 不继承终端 PATH 的典型问题
- Git 操作读写分离：读操作用 git2（零进程开销），写操作走系统 git CLI（安全、兼容性好）
- OSC 133 shell integration 用于命令检测，keystroke parsing 作为 fallback——准确度从 ~85% 提升到 ~99%

### 1.3 技术栈

```
前端：React 19 + Zustand 5 + xterm.js 6 + WebGL addon
后端：Tauri v2 + Rust（git2、which、tokio）
构建：Vite 7 + TypeScript 5.8 + pnpm
字体：Inter Variable（UI） + JetBrains Mono（终端/代码）
```

## 二、竞品格局（2026 年中）

### 2.1 直接竞品

| 竞品 | 定位 | 技术栈 | 核心差异 | 用户量级 |
|------|------|--------|---------|---------|
| **Warp** | Agentic Dev Environment | Rust 自研渲染 | 2026.4 开源（AGPLv3）；Oz 云编排支持数百并行 agent；已有 macOS + Linux + Windows(预览)；TOML 配置 | 头部玩家 |
| **cmux** | Agent 多会话管理器 | Swift 原生 + Ghostty libghostty | 专注多 agent 并发管理；状态可视化（绿/黄/红）；MCP agent 编排；Homebrew 安装 | 22k GitHub stars |
| **Wave Terminal** | 开源 AI 终端 | Electron + Go | BYOK 模型集成（OpenAI/Claude/Gemini/Ollama）；内置浏览器/SSH；跨平台 | 中量级开源 |

### 2.2 间接竞品

| 竞品 | 定位 | 与 Conduit 的关系 |
|------|------|------------------|
| **Ghostty** | 高性能终端 | Zig 构建，120fps/sub-2ms 延迟/45MB 内存；不做 AI，但被 cmux 用作渲染引擎 |
| **iTerm2** | macOS 传统终端 | 免费、稳定、插件生态；无 AI 功能但仍是 macOS 默认选择 |
| **Kitty / WezTerm / Alacritty** | 高性能终端 | 强调性能和可定制，不做 AI；用户群与 Conduit 部分重叠 |

### 2.3 Conduit vs 竞品详细对比

#### vs Warp

| 维度 | Warp | Conduit |
|------|------|---------|
| 开源协议 | AGPLv3（2026.4 开源） | 私有（GitHub 私有仓库） |
| 跨平台 | macOS + Linux + Windows(预览) | macOS only |
| AI 集成 | 自研 Oz 编排 + 内置 AI 问答 | Agent 无关（不绑定自家 AI） |
| 渲染性能 | 自研 GPU 渲染 | xterm.js + WebGL addon |
| 体积 | ~150MB | ~30MB（Tauri 优势） |
| Git 审查 | 无内置 diff 审查 | DiffPanel 三栏布局 |
| 云能力 | Oz 支持数百并行 agent | 纯本地 |

**判断：** Warp 在规模、生态和品牌上已经建立壁垒。Conduit 不应在 "通用 AI 终端" 赛道上正面竞争。

#### vs cmux

| 维度 | cmux | Conduit |
|------|------|---------|
| 多 agent 并行 | 核心卖点，无限 pane | 单次 split（最多 2 pane） |
| Agent 状态可视化 | 绿/黄/红 状态指示 + 通知 | SessionCard 状态点 + AgentStatusBar |
| Git 审查 | 无 | DiffPanel 内置 diff 查看 |
| 渲染引擎 | Ghostty libghostty（原生） | xterm.js + WebGL（Web 层） |
| 安装 | Homebrew 一键安装 | 需手动构建 |
| MCP 编排 | 支持 CMUX Agent MCP 协议 | 不支持 |

**判断：** cmux 是最直接的竞品，定位几乎相同。cmux 在 "多 agent 并行管理" 上更强；Conduit 在 "agent 改了什么" 的审查维度上更强。这个差异化是 Conduit 的生存空间。

#### vs Wave Terminal

| 维度 | Wave | Conduit |
|------|------|---------|
| 定位 | 功能丰富的 AI 终端 | Agent 协作工作台 |
| AI 集成 | BYOK 内置聊天 | 不内置 AI，只做 agent 的壳 |
| 功能广度 | 文件管理、编辑器、SSH、浏览器 | 终端 + Git diff 审查 |
| 跨平台 | macOS + Linux + Windows | macOS only |

**判断：** Wave 更偏 "全能终端"，与 Conduit 的聚焦路线不冲突。

## 三、造 Conduit 的真实动机

Conduit 不是为了在 "AI 终端" 赛道上跟 Warp/cmux 抢市场。造它的原因很直接：

> **Warp 太重了，cmux 太丑了。我就想做一个好看的、带侧栏的终端。**

这意味着 Conduit 的核心价值不是 "AI 原生"——AI agent 检测和 git diff 面板是加分项，但不是造它的理由。真正的理由是：

1. **轻量**：Tauri ~30MB vs Warp ~150MB vs Wave/Electron ~200MB+
2. **好看**：设计品味驱动——毛玻璃、强调色、Warp 风格圆形 agent 图标、精致的侧栏卡片
3. **带侧栏**：多会话管理不靠 tab 密集排列或 tmux 分屏，而是用可视化侧栏展示所有会话的实时状态

这个定位更接近 **"一个有设计品味的现代终端"** 而不是 "AI agent 编排平台"。AI 能力是锦上添花——因为你日常在终端里跑 Claude Code / Codex，所以自然需要 agent 检测和 diff 审查，但这不是唯一的使用场景。

### 3.1 这个定位的优势

- **不需要跟 Warp/cmux 正面竞争**：它们在做 agent 编排平台，Conduit 在做好看好用的终端
- **用户群更大**：不只是跑 AI agent 的人会用——任何想要一个轻量、好看、带侧栏的现代终端的开发者都是目标用户
- **维护成本可控**：不需要做云编排、MCP 协议、多 agent 并行管理这些重功能

### 3.2 从这个定位看当前的问题

| 问题 | 分析 |
|------|------|
| Agent 检测是被动的 | 合理——agent 检测是 "认出来就行"，不需要做增强体验。侧栏显示品牌图标 + 状态点已经够了 |
| 单次 Split 够不够 | 够了——2 pane 满足 95% 场景，不需要追 cmux 的无限 pane |
| DiffPanel 要不要做 Agent Review | 不一定——当前的只读 diff 审查已经比 Warp/cmux 多了一个维度，不需要做到 "agent 改动时间线" 那么重 |
| 跨平台 | 重要——既然定位是 "好看的现代终端"，用户群越大越好，Linux 支持有意义 |

## 四、功能建议（按优先级）

> 排序逻辑：Conduit 首先是一个好看、轻量、带侧栏的现代终端。AI agent 能力是加分项，不是唯一方向。优先做让终端体验更好的事。

### P0：终端体验打磨

这些是让 Conduit 作为日常终端"用起来舒服"的基础。

#### 4.1 终端搜索（⌘F）

**现状：** 已引入 `@xterm/addon-search`（在 package.json 里），但没有 UI 入口。
**方案：** ⌘F 弹出终端内搜索条（输入框 + 上/下/关闭），调用 xterm search addon。这是终端的基本功能，iTerm2/Warp/Ghostty 都有。

**工作量估计：** ~120 行（搜索条 UI + 快捷键注册 + search addon 连接）

#### 4.2 Tab 拖拽排序

**之前列为"不做"**，但既然定位是好用的侧栏终端，会话多了之后手动排序是刚需。不需要做得很复杂——侧栏 SessionCard 支持拖拽重排即可，不需要跨目录分组拖拽。

**工作量估计：** ~150 行（HTML5 drag-and-drop，不引入拖拽库）

#### 4.3 一键在外部编辑器打开

DiffPanel 里看到文件改动，点击直接跳到 VS Code / Cursor。

```
│ M  src/app/App.tsx  +12 −3   [↗]  │  ← 点击 [↗] 打开编辑器
```

**实现：** Tauri `opener` 插件 + `code --goto file:line` 或 `cursor file:line`。设置里可选默认编辑器。

**工作量估计：** ~60 行

#### 4.4 终端渲染性能

| 优化 | 做法 | 影响 |
|------|------|------|
| Scrollback | 默认从 5000 减到 3000，设置里可调 | 内存减少 ~40% |
| 大输出节流 | `onData` 加 RAF 合批，避免 agent 大量输出时卡顿 | 流畅度提升 |

### P1：设计品味提升

既然造 Conduit 的核心原因是"cmux 太丑了"，设计品味就是核心竞争力。

#### 4.5 主题系统扩展

**现状：** 只有 light / dark / system 三选一，终端配色硬编码为一套 light + 一套 dark。
**方案：** 支持 3-5 套内置主题（如 One Dark、Catppuccin、Tokyo Night、Solarized），在设置面板里以预览卡片形式选择。

**工作量估计：** ~200 行（主要是配色数据 + Settings UI）

#### 4.6 侧栏宽度可调

**现状：** 侧栏宽度固定为 CSS 变量 `--w-sidebar`。
**方案：** 侧栏右边缘可拖拽调整宽度（复用 SplitHandle 的拖拽逻辑），宽度持久化到 UI store。

**工作量估计：** ~80 行

#### 4.7 窗口模糊/透明度

**现状：** 毛玻璃效果用 CSS `backdrop-filter`，但 Tauri 的 `window.effects` 可以用原生 vibrancy。
**方案：** 用 Tauri v2 的 `window.effects.mica` (Windows) / `window.effects.underWindowBackground` (macOS) 做原生级别的毛玻璃。设置里可调透明度。

**工作量估计：** ~60 行

### P2：实用功能补齐

#### 4.8 从 Command Palette 一键启动 Agent（已砍）

该入口会把 Conduit 从“真实终端识别 CLI”推回 agent launcher / catalog。当前已从命令面板移除；用户应在真实终端中运行 CLI，Conduit 只做识别、状态和 review 辅助。

**状态：** 不进入当前产品路线。

#### 4.9 Settings Agents tab 加启动按钮（已砍）

设置页只做 CLI 路径检测，不提供启动按钮，避免把设置页做成 agent 管理平台。

**状态：** 不进入当前产品路线。

#### 4.10 DiffPanel 文件路径优化

**现状：** 只显示文件名（`Button.tsx`），同名文件分不清。
**方案：** 显示最后两级路径（`components/Button.tsx`）。

**工作量估计：** ~10 行

### P3：跨平台

#### 4.11 Linux 支持

Tauri v2 原生支持 Linux，tauri.conf.json 已配置 deb/rpm/AppImage。需要：

| 问题 | 文件 | 修复 |
|------|------|------|
| `titleBarStyle: "Overlay"` | tauri.conf.json | Linux 用系统标题栏 |
| `trafficLightPosition` | tauri.conf.json | Linux 不设置 |
| Login shell PATH 检测 | resolver/mod.rs | 已兼容（`$SHELL` fallback `/bin/zsh`），需验证 `/bin/bash` |
| CI 打包 | GitHub Actions | 需加 Linux 构建矩阵 |

#### 4.12 Windows 支持（低优先级）

| 问题 | 文件 | 修复 |
|------|------|------|
| PTY spawn | pty-bridge | 需测试 Windows ConPTY |
| `$SHELL` 环境变量 | resolver/mod.rs | Windows 无 `$SHELL`，需 fallback 到 `cmd.exe` / `powershell` |
| 路径分隔符 | 多处 | `/` vs `\` |

## 六、不建议做的功能

| 不做的事 | 理由 |
|---------|------|
| **内置 AI 聊天 / AI 问答** | Conduit 是终端不是 AI 产品。内置聊天会让它变成又一个 Warp，偏离"轻量"初衷 |
| **Agent 编排 / MCP 协议** | 这是 cmux 的方向。做编排就要做云、做多 agent 通信，复杂度爆炸 |
| **Agent stdout 结构化解析** | 需要为 11 种 agent 各写 parser，维护成本远超收益。agent CLI 的输出格式不稳定 |
| **插件系统** | 产品太早期。过早抽象会变成负担——先把核心体验做到 90 分 |
| **SSH / 远程连接** | 偏运维场景，会把产品变重。Wave 做了但也没做好 |
| **自研渲染引擎** | xterm.js + WebGL 的性能足够到 v1.0。如果未来真需要，集成 libghostty 比自研更合理 |
| **递归分栏（tmux 级 tile）** | 一次 split 覆盖 95% 场景。递归 tile 的 UI 复杂度和 bug 量呈指数增长 |
| **内置文件编辑器** | Conduit 不是 IDE。文件编辑交给外部编辑器，一键打开就够了 |
| **DiffPanel commit/push** | 之前砍掉是对的。用户在终端里 `git commit` 或让 agent commit，不需要 GUI 按钮 |
| **Agent 改动时间线** | 太重了。需要后端记录 git ref 快照 + 前端分组展示，复杂度不匹配 "轻量终端" 定位 |

## 七、战略判断

### 7.1 Conduit 的位置

Conduit 不是在跟 Warp/cmux 抢 "AI 终端" 赛道。它填的是另一个空位：

```
Warp：功能强但太重（~150MB），UI 过度设计
cmux：定位对但太丑，纯工程实用主义
Ghostty：性能极致但功能极简，没有会话管理 UI
iTerm2：免费稳定但老旧，没有现代设计语言

Conduit 的位置：轻量 + 好看 + 带侧栏 ← 这里没人做好
```

这个位置的优势是**用户群更大**——不只是跑 AI agent 的人，任何想要一个现代终端的开发者都是潜在用户。AI agent 检测 + git diff 面板是差异化加分，但不是唯一卖点。

### 7.2 建议的资源分配

| 方向 | 精力分配 | 理由 |
|------|---------|------|
| 终端基础体验（P0） | 40% | 搜索、性能、拖拽排序——让它作为日常终端可靠好用 |
| 设计品味（P1） | 30% | 主题、侧栏可调、原生毛玻璃——"好看"是造它的核心原因 |
| 实用功能（P2） | 15% | 一键启动 agent、路径优化——锦上添花 |
| 跨平台（P3） | 15% | Linux 支持——扩大用户群 |

### 7.3 核心叙事建议

当前 tauri.conf.json 里的描述：
> "Conduit — an open-source AI-native terminal emulator"

建议改为更直接的：
> "Conduit — a lightweight, beautiful terminal with sidebar"

或中文版：
> "Conduit — 轻量好看的侧栏终端"

"AI-native" 这个词在 2026 年已经被用滥了，而且容易让人拿去跟 Warp 比。"轻量好看"直接说清了 Warp 太重、cmux 太丑这两个动机。
