# Tunara vs 前沿终端：功能差距复核

> 复核日期：2026-06-22
> 复核口径：以当前代码为准，对标 Warp / Otty / Ghostty / iTerm2 / WezTerm / kitty 的官方资料。Tunara 的产品边界仍是“带智能侧栏的真实终端”，不是 agent 编排平台、聊天工具、IDE 或 Git GUI。

## 一、外部基线

| 产品 | 可借鉴方向 | 对 Tunara 的取舍 |
|------|------------|-------------------|
| Warp | Blocks、sticky command header、session restoration、command corrections、Agent/Terminal 双模式、code review panel | 借鉴“结构化终端输出”和“agent 运行可见性”；不借鉴独立 Agent Mode、云 agent、内置编辑器和编排能力 |
| Otty | 轻量侧栏、任务进度、session recovery、command palette、agent 任务队列/并行监控 | 借鉴视觉克制、运行状态、恢复；不借鉴 prompt queue、chat fork、open PR 等 agentic ADE 操作 |
| Ghostty | 原生 UI、GPU 加速、零配置、高质量 VT/主题/配置 | 借鉴终端基本功与低摩擦；Tunara 继续用 Tauri + xterm.js，不自研渲染引擎 |
| iTerm2 | split panes、搜索、shell integration、notifications、badges、captured output、annotations | 借鉴 shell integration、通知、状态标识；captured output/annotations 可作为后续候选 |
| WezTerm / kitty | 多平台、可配置、scrollback、hyperlinks、图片协议、remote control / multiplexer | 借鉴可配置和终端协议基本功；SSH、remote control、插件/脚本控制暂不做 |

## 二、当前已补齐

| 能力 | 当前实现 | 判断 |
|------|----------|------|
| 可点击 URL | `TerminalView.tsx` 使用 `WebLinksAddon`，只允许 `http:` / `https:` 后调用 `openUrl` | 已补齐，且比早期建议更安全 |
| 完成通知 / Bell | `TerminalView.tsx` 在 agent exit、命令完成和 `term.onBell` 时调用 `requestUserAttention`，受 `bellNotification` 开关控制 | 已补齐 |
| Scrollback 配置 | `ui.ts` 持久化 `scrollback`，Settings 提供 1000-20000 调整，运行时同步到 xterm | 已补齐 |
| 命令块 / Sticky header | `useTerminalBlocks` + `TerminalBlocksBar` 记录 OSC 133 命令块，支持跳转、复制完成输出、滚动时显示“当前输出”sticky chip | 已补齐轻量版；不覆盖终端内容，不进入 full-screen TUI 时强行遮挡 |
| 智能选词 / 路径选择 | xterm `wordSeparator` 保留 `/ . - :` 等路径字符，双击路径、URL、错误位置时不容易被切碎 | 已补齐基础版；不默认启用“选中即复制” |
| Captured output / 错误导航 | `registerTerminalFileLinkProvider` 识别本地 `file:line[:column]`，点击后通过现有 `open_in_editor` 跳外部编辑器 | 已补齐轻量版；只做本地文件位置，不解析编译器或 agent stdout 语义 |
| 字体连字 | `font_ligatures` 设置项控制 xterm `registerCharacterJoiner`，只 join 常见编程符号序列 | 已补齐 opt-in 版；不引入 Node/Electron 定位的 ligature addon，不为连字牺牲默认性能 |
| Agent 识别和状态 | `agent-lifecycle.ts` + shell hooks + 状态条 + 侧栏状态 | Tunara 核心差异化 |
| 只读 Git / 文件审查 | `DiffPanel` + `FileExplorer` + 外部编辑器跳转 | 符合“review 辅助、不做 Git GUI”边界 |
| 会话恢复 | Tauri store + terminal snapshot + agent resume intent schema | 仍不是完整 agent 对话恢复，但已满足基础终端恢复 |

## 三、仍缺但值得做

| 优先级 | 缺口 | 为什么值得做 | 建议 |
|--------|------|--------------|------|
| P3 | Inline image / graphics protocols | 现代终端基本功之一，但 xterm.js 生态和 Tunara 目标用户不强依赖 | 暂不做，除非真实用户场景出现 |

## 四、需要 /think 再决策

这些方向有产品边界风险，不能直接做：

| 方向 | 风险 | 决策问题 |
|------|------|----------|
| 全局热键 / quick terminal | 可能变成系统级常驻工具，涉及权限、窗口管理和多屏体验 | Tunara 是否要做“随手召出终端”，还是保持项目窗口型应用 |
| 命令纠错 | Warp 的 command corrections 有价值，但会引入建议系统和错误分类规则 | 是否接受非 AI 的本地规则纠错；是否允许把建议插入终端输入行 |
| prompt queue / send to chat / fork chat | Otty/Warp 的 agentic ADE 能力，但明确接近聊天/编排 | 默认不做；只有用户明确改产品边界才讨论 |
| remote control / multiplexer / SSH | WezTerm/kitty 强项，但 AGENTS 已明确不做 SSH 远程和插件系统 | 默认不做 |

## 五、明确不做

- 独立 Agent Mode、agent catalog、新建 Agent 弹层。
- 内置 AI 聊天、BYOK、MCP、云 agent、agent 编排。
- DiffPanel 里的 stage / commit / push。
- 内置文件编辑器、LSP、IDE 化 file tree。
- 递归无限分屏、自由 tile layout。
- SSH 远程、remote control、插件系统、自研渲染引擎。

## 六、本轮结论

Tunara 的主要缺口已经从“基础终端能力缺失”收敛到“图形协议”等体验增强项。当前最值得继续做的是：

1. 图形协议 / inline image：继续保持 P3，等真实用户场景出现后再评估。

本轮已经补齐 command block 的运行中状态、智能选词基础版、不遮挡终端内容的 sticky command header 轻量版、`file:line[:column]` 错误导航，以及 opt-in 字体连字。
