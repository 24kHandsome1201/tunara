## 1. ⌘+/- 快捷调字号 + ⌘0 恢复默认

**优先级**：高，终端标配快捷键，缺失显眼
**工作量**：0.5 天
**涉及文件**：`src/app/useKeybindings.ts`、`src/state/ui.ts`

### 现状

- 字号只能通过 Settings 面板手动 +/- 调整
- `useKeybindings.ts` 没有处理 `=`/`-`/`0` 键
- `setFontSize` setter 没有 clamp，但 Settings UI 限制了 10-22 范围

### 实现

在 `useKeybindings.ts` 的 `onKey` 处理器中，匹配 `⌘=`（加大）、`⌘-`（缩小）、`⌘0`（恢复默认 14px）：

```
useKeybindings.ts：在 `if (k === "k")` 块之后追加：

} else if (k === "=" || k === "+") {
  e.preventDefault();
  const ui = useUIStore.getState();
  ui.setFontSize(Math.min(22, ui.fontSize + 1));
} else if (k === "-") {
  e.preventDefault();
  const ui = useUIStore.getState();
  ui.setFontSize(Math.max(10, ui.fontSize - 1));
} else if (k === "0") {
  e.preventDefault();
  useUIStore.getState().setFontSize(14);
}
```

同时给 `ui.ts` 的 `setFontSize` 加 clamp 防御：

```
setFontSize: (fontSize) => set({ fontSize: Math.max(10, Math.min(22, fontSize)) }),
```

### 验证

- ⌘= 字号 +1，到 22 停
- ⌘- 字号 -1，到 10 停
- ⌘0 恢复到 14px
- 终端 xterm.js 实时跟随（TerminalView.tsx:585 已有 `term.options.fontSize = fontSize` 响应）
- 不影响 input/textarea 里的浏览器默认缩放行为（`isEditableTarget` 已过滤）

## 2. ⌘1-9 按位置切会话

**优先级**：高，终端标配快捷键
**工作量**：0.5 天
**涉及文件**：`src/app/useKeybindings.ts`

### 现状

- 会话切换只能通过侧栏点击或 Command Palette 搜索
- `useKeybindings.ts` 没有处理数字键

### 实现

在 `useKeybindings.ts` 的 `onKey` 处理器中，匹配 `⌘1` 到 `⌘9`：

```
useKeybindings.ts：在字号快捷键块之后追加：

} else if (k >= "1" && k <= "9") {
  e.preventDefault();
  const idx = parseInt(k) - 1;
  const sessions = useSessionsStore.getState().sessions;
  if (idx < sessions.length) {
    useSessionsStore.getState().setActive(sessions[idx].id);
  }
}
```

### 语义说明

- index 按 sessions 数组顺序（即侧栏从上到下的展示顺序，跨目录分组展平）
- ⌘9 不是切到第 9 个，而是切到最后一个（和 Chrome/iTerm 行为一致）

```
// ⌘9 特殊处理：切到最后一个
} else if (k === "9") {
  e.preventDefault();
  const sessions = useSessionsStore.getState().sessions;
  if (sessions.length > 0) {
    useSessionsStore.getState().setActive(sessions[sessions.length - 1].id);
  }
}
```

### 验证

- ⌘1 切到第一个会话
- ⌘3 切到第三个会话（如果存在）
- ⌘9 切到最后一个会话
- 超出范围的数字无反应（不崩溃）
- 分屏模式下正常切换 active（ensureSessionVisibleInSplit 已接入 setActive）

## 3. Staged vs Unstaged 区分显示（审查面板）

**优先级**：中，提升审查面板从"信息展示"到"可操作"的关键一步
**工作量**：2-3 天（后端 + 前端）
**涉及文件**：

| 层 | 文件 | 改动 |
|---|------|------|
| 后端 | `src-tauri/src/modules/git/mod.rs` | 拆分 staged / unstaged diff |
| 前端桥接 | `src/modules/git/git-bridge.ts` | 新增 `stage` 字段类型 |
| 前端 UI | `src/ui/DiffPanel.tsx` | 分区显示 + 设计（需 /design） |

### 现状

- `git_diff` 使用 `diff_tree_to_workdir_with_index` 合并了 staged + unstaged，前端无法区分
- `git_status` 返回 `files` 列表但没有 staged 标记
- 后端 `commit.rs` 有 `git_add_args` 但注释说明"Git commit/push are no longer exposed through Tauri IPC"

### 后端实现

`src-tauri/src/modules/git/mod.rs`：`git_status` 返回的 `FileStatus` 增加 `stage` 字段：

```rust
#[derive(Serialize)]
pub struct FileStatus {
    pub path: String,
    pub status: String,    // "M" / "A" / "D" / "?"
    pub stage: String,     // "staged" / "unstaged" / "both" / "untracked"
}
```

解析 `git status --porcelain=v1` 的两列状态码：
- 第 1 列有值 + 第 2 列无值 → `staged`
- 第 1 列无值 + 第 2 列有值 → `unstaged`
- 两列都有值 → `both`
- `??` → `untracked`

`git_diff` 返回的 `FileDiff` 增加可选 `stage` 字段（和 status 保持一致）。

### 前端桥接

`src/modules/git/git-bridge.ts`：`FileChange` 类型增加：

```typescript
export interface FileChange {
  path: string;
  status: string;
  stage: "staged" | "unstaged" | "both" | "untracked";
}
```

### 前端 UI：需要 /design 设计

`src/ui/DiffPanel.tsx`：当前文件列表需要分成两个视觉分区：

- **Staged Changes**：绿色左边条或类似视觉标记
- **Unstaged Changes**：默认色
- **Untracked**：灰色虚线或 "?" 标记

**设计要求**：
- 分区 header 样式（"已暂存 · 3 个文件" / "未暂存 · 5 个文件"）
- 每个文件行需要区分 staged/unstaged 的视觉差异
- 折叠/展开分区的交互（如果文件较多）
- 和现有 DiffPanel 的文件行样式（diff-file-row）保持一致
- 保留外部编辑器跳转按钮的 hover 显示

**不做（本期）**：
- Stage/Unstage 操作按钮（后续可加）
- Commit UI（后续可加）
- 单文件 stage 的 hunk-level 选择

### 验证

- 有 staged 文件时显示分区
- 全部 unstaged 时不显示 staged 分区 header
- `git add` 一个文件后刷新，该文件从 unstaged 移到 staged
- 同一文件部分 staged 时显示在 `both` 区
- untracked 文件正确归类
